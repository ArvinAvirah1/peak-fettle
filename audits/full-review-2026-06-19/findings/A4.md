### [P0] A4-01 — exercise-library: modal set-history always calls REST, breaks free users

- **File:** `mobile/app/exercise-library.tsx:594–611`
- **Problem:** `ExerciseDetailModal` always fetches personal set history via `apiClient.get('/sets', ...)` with no `isLocalFirst` branch. For free users, this REST call fails (no server-side personal data) — the "Personal Best" and "Volume History" sections permanently show `setsError: 'Could not load history.'`. Free users can never see their own exercise history in this modal.
- **Evidence:**
```ts
apiClient
  .get<{ sets: SetRecord[] }>('/sets', {
    params: { exercise_id: exercise.id, limit: SET_HISTORY_LIMIT },
  })
  .then((res) => {
    if (!cancelled) setSets(res.data.sets ?? []);
  })
  .catch(() => {
    if (!cancelled) setSetsError('Could not load history.');
  })
```
- **Invariant/Rubric:** Invariant 1 (local-first tiering). Free users make no personal REST calls; `/sets?exercise_id=…` is personal data.
- **Suggested direction:** Add an `isLocalFirst(user)` branch in the `ExerciseDetailModal` effect (already has `useAuth`). For free users, query `localDb` with `SELECT * FROM sets WHERE exercise_id = ? AND kind = 'lift' ORDER BY logged_at DESC LIMIT 100`. Apply the same `setKg()`/`epleyE1RM()` helpers already in scope.
- **Confidence:** HIGH

### [P0] A4-02 — progress.tsx server path: weight_raw/8 ignores weight_kg, corrupts PRs and volume

- **File:** `mobile/app/progress.tsx:219, 238–249`
- **Problem:** `fetchProgressData()` (the Pro/server path) uses `s.weight_raw / 8` directly to compute volume and e1RM. The `ApiSet` type declares `weight_raw: number` but no `weight_kg` field. If the server returns `weight_kg` (exact kg, as it does for newer sets), that column is ignored and volume/PR values are computed from the lossy `weight_raw` only. This silently corrupts all analytics for Pro users with recent exact-kg sets. The local path (`fetchLocalProgressData`) correctly uses `COALESCE(weight_kg, weight_raw/8.0)` via its `kgOf()` helper — the server path must do the same.
- **Evidence:**
```ts
// progress.tsx:219 — fetchProgressData() volume loop:
const vol = (s.weight_raw / 8) * s.reps;
// progress.tsx:238 — epley helper:
function epley(weightRaw: number, reps: number): number {
  const kg = weightRaw / 8;
  return kg * (1 + reps / 30);
}
// Called at line 238: const e1rm = epley(s.weight_raw, s.reps);
```
- **Invariant/Rubric:** Invariant 2 (weight = exact kg). Server path must read `weight_kg` when present and fall back to `weight_raw/8` — matching the local path's `kgOf()` pattern.
- **Suggested direction:** Extend `ApiSet` to include `weight_kg?: number`; add a `setKgPro(s)` helper mirroring `kgOf()`: `s.weight_kg ?? s.weight_raw / 8`. Use it in both the volume accumulator and the PR `epley()` call.
- **Confidence:** HIGH

### [P0] A4-03 — exercise-library: goal weight saved without unit conversion (lbs users store display value as kg)

- **File:** `mobile/app/exercise-library.tsx:561–571, 847–854`
- **Problem:** `handleSaveGoal` calls `setExerciseGoal(exercise.id, w, r, ...)` where `w = parseFloat(goalWeight)`. The `goalWeight` text input is labelled "Weight (kg)" but `unitPref` is read from `user?.unit_pref`. If the user's preference is `lbs`, they will type e.g. `225` (lbs), and that value is passed directly as `targetWeightKg` — storing 225 kg in `exercise_goals.target_weight_kg`. This is the exact lbs-as-kg bug the repo documented after a prior missed conversion.
- **Evidence:**
```ts
const w = parseFloat(goalWeight);   // user typed lbs value for a lbs-pref user
const r = parseInt(goalReps, 10);
// ...
setExerciseGoal(exercise.id, w, r, exercise.name)  // w treated as kg — BUG
```
```ts
// Input placeholder always says "kg" regardless of unitPref:
placeholder="Weight (kg)"
```
- **Invariant/Rubric:** Invariant 2 (weight = exact kg). Display→storage conversion must use `displayToKg(w, unitPref)` before calling `setExerciseGoal`.
- **Suggested direction:** Import `displayToKg` and `parseWeightInput` from `units.ts`. In `handleSaveGoal`, replace `parseFloat(goalWeight)` with `parseWeightInput(goalWeight)` and convert via `displayToKg(w, unitPref)` before passing to `setExerciseGoal`. Change the placeholder to show the active unit label. Similarly, prefill `goalWeight` via `kgToInputValue(goal.target_weight_kg, unitPref)` when editing an existing goal (line 950 currently passes raw kg string).
- **Confidence:** HIGH

### [P1] A4-04 — workout-day.tsx: setVolumeKg and bestSetIds use weight_raw only, ignores weight_kg for local sets

- **File:** `mobile/app/workout-day.tsx:121–123, 660–665`
- **Problem:** `setVolumeKg()` unconditionally reads `s.weight_raw` (dividing by 8), and `bestSetIds` computation calls `computeE1rm(s.weight_raw, s.reps)`. For local sets fetched by `fetchLocalDayData`, `weight_raw` is synthetically set to `r.weight_kg * 8` (line 322), so the round-trip is correct. However for the server path (`fetchDayData`), server may return only `weight_kg` (no `weight_raw`), making `s.weight_raw` `undefined` — both functions then compute `undefined / 8 = NaN` and volume shows as `NaN` / best-set highlighting breaks silently.
- **Evidence:**
```ts
function setVolumeKg(s: ApiSet): number {
  if (s.kind !== 'lift' || !s.weight_raw || !s.reps) return 0;
  return rawToKg(s.weight_raw) * s.reps;  // ignores weight_kg
}
// bestSetIds:
if (s.kind === 'lift' && s.weight_raw && s.reps) {
  const e = computeE1rm(s.weight_raw, s.reps);  // weight_raw may be undefined for server sets
```
- **Invariant/Rubric:** Invariant 2 (weight = exact kg). Rubric P0 data integrity.
- **Suggested direction:** Add a `resolvedWeightRaw(s: ApiSet): number` helper that returns `s.weight_kg != null ? s.weight_kg * 8 : (s.weight_raw ?? 0)`. Use it in `setVolumeKg`, `computeE1rm` call sites, and `SetRow`. The local path already does this synthesis at line 322 — replicate it at the helper level.
- **Confidence:** HIGH

### [P1] A4-07 — workout-history/workout-day: TSC TS2345/TS18048 on destructured date array elements

- **File:** `mobile/app/workout-history.tsx:70`, `mobile/app/workout-day.tsx:138`
- **Problem:** Both files destructure `dateStr.split('-').map(Number)` and use the elements directly as `new Date(year, month - 1, day)`. TypeScript flags `month` as `number | undefined` (TS18048) and `month - 1` as `TS2345`. This is a real risk if `dateStr` is empty string or malformed — `undefined - 1 = NaN` yields an `Invalid Date`, which would silently propagate as `NaN` date comparisons.
- **Evidence:**
```ts
// workout-history.tsx:69-70:
const [year, month, day] = dateStr.split('-').map(Number);
return new Date(year, month - 1, day);   // TS18048: month possibly undefined
// workout-day.tsx:137-138:
const [year, month, day] = dateStr.split('-').map(Number);
const d = new Date(year, month - 1, day);  // same
```
- **Invariant/Rubric:** Rubric P1 null-safety (confirmed TS18048/TS2345 in tsc output).
- **Suggested direction:** Add a null-coalescing guard: `new Date(year ?? 1970, (month ?? 1) - 1, day ?? 1)` — same pattern used in `routine-history.tsx:42` which already has this fix.
- **Confidence:** HIGH

### [P2] A4-08 — exercise-library: debounce timer not cleared on unmount (floating timer)

### [P2] A4-09 — progress.tsx: `isoWeek()` uses `new Date(dateStr)` with UTC parse → off-by-one day in local time

- **File:** `mobile/app/progress.tsx:132–139`
- **Problem:** `isoWeek(dateStr)` calls `new Date(dateStr)` on a `started_at` ISO timestamp (correct) but for the local path it receives `w.day_key` which is a `YYYY-MM-DD` string. When parsed as `new Date('2026-06-01')`, JS treats it as UTC midnight → in timezones west of UTC the date resolves to the previous day locally. This shifts the week bucket, causing a workout to appear in the wrong ISO week.
- **Evidence:**
```ts
function isoWeek(dateStr: string): string {
  const d = new Date(dateStr);  // YYYY-MM-DD parsed as UTC → local day shift
  const jan4 = new Date(d.getFullYear(), 0, 4);
  // ...
}
// Local path (line 303):
const wk = isoWeek(w.day_key);  // day_key = YYYY-MM-DD — UTC parse bug
```
- **Invariant/Rubric:** Rubric P2 — incorrect date handling under timezone edge case.
- **Suggested direction:** For `YYYY-MM-DD` strings, parse local-time with the same split pattern used by `parseLocalDate()` in `workout-history.tsx`: `const [y,m,d] = dateStr.split('-').map(Number); const d = new Date(y, m-1, d)`. Apply it in `fetchLocalProgressData`'s `isoWeek()` call path or add a `localIsoWeek(dayKey)` variant.
- **Confidence:** MED (only affects users in UTC-offset timezones; most common case is fine)

### [P3] A4-11 — one-rm.tsx: weight input interprets display value directly as 1RM (no unit conversion bug, but UX label risk)

- **File:** `mobile/app/one-rm.tsx:22, 28, 122`
- **Problem:** The 1RM calculator takes user input as the weight for a recent set, displays the result as `{oneRm.toFixed(1)} {unitLabel}`, and calls `estimate1Rm(w, r, formula)` directly. `w = parseFloat(weight)` is the raw display value (could be lbs). `estimate1Rm` treats this value as-is; since the output unit is also labeled `unitLabel`, the math is internally consistent (lbs in → lbs out). However the label `unitLabel` is `'lb'` for lbs users but `estimate1Rm` and `weightForReps` are pure numeric — they do not store anything. This is a standalone calculator with no storage writes, so no weight invariant is violated. Noting for completeness: if a future change stores the result, conversion will be needed.
- **Evidence:**
```ts
const unitLabel = (user?.unit_pref ?? 'kg') === 'lbs' ? 'lb' : 'kg';
const w = parseFloat(weight) || 0;
// ...
{oneRm.toFixed(1)} {unitLabel}  // purely display — no storage
```
- **Invariant/Rubric:** Rubric P3 — no current data loss; flag for future storage paths.
- **Suggested direction:** No change needed now. If `oneRm` is ever saved to `sets`, apply `displayToKg(oneRm, unitPref)` before storage.
- **Confidence:** HIGH (no bug today)
