### [P0] HOME-01 — `getPercentile` `useEffect` has no cancel flag; `setState` after unmount is possible

- **File:** `mobile/app/(tabs)/index.tsx:485–500`
- **Problem:** The `useEffect` that calls `getPercentile()` (Pro path only) fires on every `user?.is_paid` change and awaits a network round-trip. There is no `cancelled` flag and no cleanup return. If the component unmounts (e.g., a rapid tab switch) while the request is in flight, `setBestPercentile(Math.max(...values))` fires on the unmounted component. In React Native's concurrent renderer this is a stale-closure update that can silently corrupt state in a sibling that recycled the slot.  Additionally, `loadPlan` (lines 462–476) inside a `useCallback` + `useEffect` has the same pattern — both the `getPlans()` and the chained `getPlan(plans[0].id)` await can settle after unmount, calling `setPlan`, `setPlanLoading` on a dead component.

- **Evidence:**
```ts
useEffect(() => {
  if (!user?.is_paid) return;
  getPercentile()
    .then((resp) => {
      const values = resp.rankings
        .map((r) => r.percentile)
        .filter((v): v is number => v !== null);
      if (values.length > 0) setBestPercentile(Math.max(...values)); // no cancel guard
    })
    .catch((err: unknown) => { /* … */ });
}, [user?.is_paid]); // no cleanup
```

- **Invariant/Rubric:** P0 — Async race: `useEffect` with no cleanup/cancel flag; `setState` after unmount.
- **Suggested direction:** Add a `let cancelled = false` flag; return `() => { cancelled = true; }` from the effect; guard each `.then` branch with `if (cancelled) return;`. Apply the same pattern to `loadPlan` (the `useEffect(() => { loadPlan(); }, [loadPlan])` block also needs a cancel guard, or `loadPlan` itself should accept an `AbortSignal`).
- **Confidence:** HIGH

### [P1] HOME-02 — Three `Modal` bottom-sheets missing Dynamic Island `paddingTop` guard (Invariant 3)

- **File:** `mobile/app/(tabs)/index.tsx:777–842` (Streak Detail), `849–889` (Forgot-something prompt), `893–934` (Today's lifts list)
- **Problem:** All three Modals present as bottom-sheets (`justifyContent: 'flex-end'`) and slide up to fill the screen. Their inner `View` (`styles.modalSheet`) has a fixed `padding: 24` but zero `paddingTop`-from-insets. When the sheet is tall enough to reach the status-bar area (which is guaranteed when `maxHeight: '80%'` is set on the third modal), the Dynamic Island / notch on iPhone 14+ will overlap the sheet header. `SafeAreaView` inside a `<Modal>` does not propagate insets (the known invariant). The `useSafeAreaInsets()` hook is imported in `_layout.tsx` but never used inside `index.tsx`; there is no import of it here.
- **Evidence:**
```ts
// styles.modalSheet (line 1520)
modalSheet: {
  padding: 24,
  paddingBottom: 40,
  // ← no paddingTop: Math.max(insets.top, 12)
},
// No useSafeAreaInsets() anywhere in index.tsx
```
- **Invariant/Rubric:** P1 — Invariant 3 (SafeArea in Modal). Affects all three modal sheets.
- **Suggested direction:** Import `useSafeAreaInsets` in `HomeScreen`, read `insets.top`, and pass `paddingTop: Math.max(insets.top, 12)` as an inline style override on the first child `View` inside each Modal (not on the overlay). The shared `styles.modalSheet` should NOT carry the inset — it varies per device.
- **Confidence:** HIGH

### [P1] HOME-03 — TSC TS2532 `plans[0]` possibly undefined (line 468) + TS2769 `new Date(undefined)` (lines 626–627)

- **File:** `mobile/app/(tabs)/index.tsx:468, 626, 627`
- **Problem (a — line 468):** `plans[0].id` is accessed inside `if (plans.length > 0)`, which is logically safe, but TypeScript's strict-nullchecks mode does not narrow array element access: `plans[0]` is still typed as `Plan | undefined`. This is a real TS2532 error (confirmed in `00-tsc.txt`). Runtime risk is minimal due to the length guard, but the compiler is correct — a `tsconfig` that tightens `noUncheckedIndexedAccess` would break here.
- **Problem (b — lines 626–627):** `sortedKeys[i - 1]` and `sortedKeys[i]` are `string | undefined` in strict mode. `new Date(undefined)` is `new Date(NaN)`, meaning `prev.getTime()` and `curr.getTime()` both return `NaN`, `diffDays` = `NaN`, the `diffDays === 1` branch is never taken, and `longestStreak` always returns `1` when `history.length > 0` and the loop runs at least once — a silent wrong value shown to the user in the Streak Detail sheet.
- **Evidence:**
```ts
// line 467–469
if (plans.length > 0) {
  const detail = await getPlan(plans[0].id);  // TS2532: plans[0] possibly undefined
}
// lines 625–629
const prev = new Date(sortedKeys[i - 1]);  // TS2769: string|undefined not assignable
const curr = new Date(sortedKeys[i]);       // TS2769: string|undefined not assignable
const diffDays = Math.round(
  (curr.getTime() - prev.getTime()) / (1000 * 60 * 60 * 24)
);
```
- **Invariant/Rubric:** P1 — null-safety (TS2532 / TS2769 confirmed in tsc baseline); silent wrong value for `longestStreak`.
- **Suggested direction:** (a) Use `plans.at(0)?.id` or `const first = plans[0]; if (first)`. (b) Add non-null assertions (`sortedKeys[i - 1]!`) or check `const prev = sortedKeys[i - 1]; if (!prev) continue;` — the string is guaranteed populated because the array is built from `history.map(…)`, but tsc needs the explicit narrowing.
- **Confidence:** HIGH

### [P1] HOME-04 — `loadPlan` missing from `handleRefresh` dependency path; stale plan on Pro

- **File:** `mobile/app/(tabs)/index.tsx:647–651`
- **Problem:** `handleRefresh` is declared as an `async` function (not `useCallback`), so it is recreated every render. More importantly, it calls `loadPlan()` — but `loadPlan` is a `useCallback` that returns early when `!user?.is_paid`. If the user's plan has been regenerated server-side between two manual refreshes (pull-to-refresh), the `loadPlan` call runs correctly, but it is not wrapped in a `cancelled` guard (see HOME-01). Additionally, `handleRefresh` does not reset `bestPercentile` or re-trigger the percentile effect — the "Best Rank" stat card can show a stale value after a refresh that would logically reload rankings. This is a P1 state-consistency bug for Pro users.
- **Evidence:**
```ts
const handleRefresh = async (): Promise<void> => {
  setRefreshing(true);
  await Promise.all([refetch(), loadPlan()]);
  setRefreshing(false);
  // ← bestPercentile not reloaded; percentile useEffect won't re-fire
  //   because user?.is_paid hasn't changed.
};
```
- **Invariant/Rubric:** P1 — stale state after refresh; `handleRefresh` is not `useCallback` so recreated every render (minor perf).
- **Suggested direction:** Either (a) expose a `reloadPercentile` helper from the percentile effect and call it from `handleRefresh`, or (b) add a `refreshKey` counter to the percentile `useEffect` dep array and increment it in `handleRefresh`. Wrap `handleRefresh` in `useCallback`.
- **Confidence:** MED (the percentile not refreshing is confirmed; whether this matters depends on how often rankings change between refreshes)

### [P1] ROUTINES-01 — `showToast` `setTimeout` not cleaned up on unmount

- **File:** `mobile/app/(tabs)/routines.tsx:220–228`
- **Problem:** `showToast` sets a 1800 ms `setTimeout` that calls `setToast(null)` after the animation. There is no `clearTimeout` on unmount. If the user navigates away from the Routines tab during a toast (e.g., rapidly switches tab), the callback fires on the unmounted component and calls `setToast` on dead state. This is a standard P1 setState-after-unmount via a floating timer.
- **Evidence:**
```ts
const showToast = useCallback((msg: string) => {
  setToast(msg);
  RNAnimated.timing(toastOpacity, { toValue: 1, duration: 180, useNativeDriver: true }).start();
  setTimeout(() => {               // ← no ref stored, no cleanup
    RNAnimated.timing(toastOpacity, { toValue: 0, duration: 220, useNativeDriver: true }).start(
      () => setToast(null),        // ← setState fires after unmount
    );
  }, 1800);
}, [toastOpacity]);
```
- **Invariant/Rubric:** P1 — floating promise / uncleaned timer; `setState` after unmount.
- **Suggested direction:** Store the timer in a `useRef<ReturnType<typeof setTimeout>>`, and clear it in a `useEffect` cleanup: `return () => { clearTimeout(toastTimerRef.current); }`. Also stop the in-flight `RNAnimated.timing` (call `.stop()` on the animation reference if needed).
- **Confidence:** HIGH

### [P2] HOME-05 — `prsThisWeek` `useMemo` cutoff uses `setDate` mutation — incorrect in edge cases

- **File:** `mobile/app/(tabs)/index.tsx:420–434`
- **Problem:** `const cutoff = new Date(); cutoff.setDate(cutoff.getDate() - 6)` mutates `cutoff` in place and then calls `toDateKey(cutoff)`. This is the same pattern used in `recentDays` (line 438–442). Both are inside `useMemo` calls and produce stable values. The pattern is safe here since `cutoff` is a local variable and the mutation is immediately consumed. However, comparing against the `longestStreak` logic (which sorts string keys and then uses `new Date(key)` that may receive `undefined`), this is a low-risk inconsistency. The real issue is that `prsThisWeek` counts sets in the window `cutoffKey <= day_key` (i.e., today minus 6 days = 7 calendar days), but the label in the UI says "PRs this week" — which a user interprets as the current ISO week (Mon–Sun). The window is a rolling 7 days, not the ISO week. This is a silent display inaccuracy.
- **Evidence:**
```ts
const prsThisWeek = useMemo(() => {
  const cutoff = new Date();
  cutoff.setDate(cutoff.getDate() - 6);       // rolling 7 days, not ISO week
  const cutoffKey = toDateKey(cutoff);
  // ...
}, [history]);
// StatCard label: "PRs this week"
```
- **Invariant/Rubric:** P2 — display inaccuracy; misleading label.
- **Suggested direction:** Either rename the label to "PRs (7 days)" to match the rolling window, or compute the actual ISO-week start (Monday at midnight local) as the cutoff.
- **Confidence:** HIGH (the label vs logic mismatch is certain; whether this counts as a bug or a product decision is a call for the founder)

### [P2] HOME-06 — `handleRefresh` is not memoized; `ScrollView RefreshControl` receives a new function every render

- **File:** `mobile/app/(tabs)/index.tsx:647–651`
- **Problem:** `handleRefresh` is a plain `async` arrow function declared inside the component body with no `useCallback`. Every render recreates it, which means `RefreshControl`'s `onRefresh` prop changes reference on every render. For a `ScrollView` this causes no correctness issue, but it does cause React to diff and patch the native `RefreshControl` on every render — unnecessary work on a heavy screen that re-renders frequently (history updates, streak ticks, etc.).
- **Evidence:**

```ts
const handleRefresh = async (): Promise<void> => {  // no useCallback
  setRefreshing(true);
  await Promise.all([refetch(), loadPlan()]);
  setRefreshing(false);
};
```
- **Invariant/Rubric:** P2 — unnecessary re-renders; missing `useCallback`.
- **Suggested direction:** Wrap in `useCallback([refetch, loadPlan])`.
- **Confidence:** HIGH

### [P2] ROUTINES-02 — `GestureHandlerRootView` nested inside `SafeAreaView` rather than wrapping the root

- **File:** `mobile/app/(tabs)/routines.tsx:511–512`
- **Problem:** The component renders `<SafeAreaView><GestureHandlerRootView>…</GestureHandlerRootView></SafeAreaView>`. RNGH's docs require `GestureHandlerRootView` to be at the root of the app or at least the root of the navigation tree — not nested inside an arbitrary wrapper on a single screen. The `_layout.tsx` does NOT wrap with `GestureHandlerRootView` at the tab root level. On iOS this means gesture events from `ReanimatedSwipeable` inside `RoutinesPage` propagate through two gesture recognizer roots: the one in this component and any parent provided by expo-router's root layout. This can cause gesture conflicts (swipe to dismiss the tab navigator interfering with the row swipe). Additionally, nesting inside `SafeAreaView` means `GestureHandlerRootView` doesn't fill its parent consistently since `SafeAreaView` edges insets can shrink the layout area.
- **Evidence:**
```ts
// routines.tsx:511–512
return (
  <SafeAreaView style={[styles.root, { backgroundColor: c.bgPrimary }]} edges={['top']}>
    <GestureHandlerRootView style={styles.root}>
      {/* ReanimatedSwipeable rows live in here */}
```
- **Invariant/Rubric:** P2 — RNGH gesture conflicts; incorrect `GestureHandlerRootView` placement.
- **Suggested direction:** Move `GestureHandlerRootView` to the app root (`mobile/app/_layout.tsx`, wrapping the entire `<Stack>`). Remove it from `RoutinesPage`. This is the approach recommended by RNGH docs and aligns with the CLAUDE.md invariant (`GestureHandlerRootView must wrap the app root`).
- **Confidence:** HIGH

### [P3] LAYOUT-01 — `AnimatedTabIcon` `useEffect` missing `scale` in deps array

- **File:** `mobile/app/(tabs)/_layout.tsx:54–56`
- **Problem:** The `useEffect` in `AnimatedTabIcon` that updates the Reanimated shared value has `[focused]` in its deps array but not `scale`. Because `scale` is a Reanimated `SharedValue` (a ref-like object, not a plain state value), its reference never changes and excluding it from deps is harmless — Reanimated shared values are mutable containers. However, the linter (if `react-hooks/exhaustive-deps` is enabled) will flag this as a missing dep, and reviewers unfamiliar with Reanimated might add `scale` and trigger unnecessary re-runs. A comment explaining why `scale` is intentionally excluded would prevent future confusion.
- **Evidence:**
```ts
React.useEffect(() => {
  scale.value = withSpring(focused ? 1.15 : 1.0, { damping: 10, stiffness: 200 });
}, [focused]); // scale is a SharedValue ref — intentionally omitted
```
- **Invariant/Rubric:** P3 — maintainability; missing explanatory comment.
- **Suggested direction:** Add `// scale is a Reanimated SharedValue (stable ref) — intentionally omitted from deps` inline.
- **Confidence:** HIGH
