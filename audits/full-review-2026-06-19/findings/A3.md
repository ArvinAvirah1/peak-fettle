### [P1] PLANS-01 — `PlanDetailModal` uses `SafeAreaView` inside `<Modal>` without direct `paddingTop` inset (Invariant 3)

- **File:** `mobile/app/(tabs)/plans.tsx:460–627` (modal open at :460, problematic header at :468)
- **Problem:** `PlanDetailModal` wraps its content in `<SafeAreaView edges={['top','bottom']}>` inside a React Native `<Modal>`. Per Invariant 3, `SafeAreaView` does NOT reliably push content below the Dynamic Island inside a `<Modal>`, so the header ("Plan" title + Done button) can render under the island on iPhone. The sibling `LocalPlanModal` (line 660–664) correctly applies `paddingTop: Math.max(insets.top, 12)` directly to the header `<View>` — `PlanDetailModal` was never updated to match.
- **Evidence:**
  ```tsx
  // plans.tsx:460–468 — PlanDetailModal
  <Modal visible animationType="slide" presentationStyle="pageSheet" onRequestClose={onClose}>
    <SafeAreaView style={[detailStyles.container, { backgroundColor: theme.colors.bgPrimary }]} edges={['top', 'bottom']}>
      {/* Header */}
      <View style={[detailStyles.header, { borderBottomColor: theme.colors.bgSecondary }]}>
        {/* ← no paddingTop: Math.max(insets.top, 12) here */}
  ```
  ```tsx
  // plans.tsx:660–664 — LocalPlanModal CORRECT pattern
  <View style={[
    detailStyles.header,
    { borderBottomColor: theme.colors.bgSecondary, paddingTop: Math.max(insets.top, 12) },
  ]}>
  ```
- **Invariant/Rubric:** Invariant 3 (Safe-area in Modal); P1 (wrong behavior on iPhone with Dynamic Island — content hidden under notch).
- **Suggested direction:** Add `const insets = useSafeAreaInsets()` to `PlanDetailModal` (it doesn't have one), then apply `paddingTop: Math.max(insets.top, 12)` to the `detailStyles.header` `<View>` inline, exactly as `LocalPlanModal` does on line 664.
- **Confidence:** HIGH

### [P1] RANKINGS-02 — `OverallPercentileCard` median computation has confirmed TS2532/TS2345 crash risk under `noUncheckedIndexedAccess` (TSC-confirmed)

- **File:** `mobile/app/(tabs)/rankings.tsx:605–611`
- **Problem:** Three TSC errors flagged in `00-tsc.txt` (`rankings.tsx(608,10): TS2532`, `(608,28): TS2532`, `(611,36): TS2345`) are real. With `noUncheckedIndexedAccess: true` (confirmed in `mobile/tsconfig.json`), `values[mid - 1]` and `values[mid]` are typed `number | undefined` even after the `.filter((v): v is number => v != null)` guard, because the type narrowing from filter does not narrow individual index accesses under this flag. The even-length branch `(values[mid - 1] + values[mid]) / 2` will therefore produce `NaN` if TypeScript is correct and the runtime value is actually undefined, and `Math.round(NaN)` silently renders as "NaN%" in the UI. At runtime the filter guarantees non-undefined values, but the type system cannot verify it, and a future refactor of the filter could break the invariant silently.
- **Evidence:**
  ```ts
  // rankings.tsx:605–611
  const mid = Math.floor(values.length / 2);
  const median =
    values.length % 2 === 0
      ? (values[mid - 1] + values[mid]) / 2   // TS2532 × 2: both operands possibly undefined
      : values[mid];                            // also possibly undefined
  const medianRounded = Math.round(median);    // TS2345: median is number | undefined
  ```
- **Invariant/Rubric:** P1 — null-safety (TS2532 / TS2345); confirmed TSC error in scope.
- **Suggested direction:** Add explicit undefined guards: `const lo = values[mid - 1] ?? 0; const hi = values[mid] ?? 0;` for the even branch, and `const odd = values[mid] ?? 0;` for the odd branch. Alternatively, index with a non-null assertion (`values[mid]!`) only after confirming the length-guard makes it safe. Either approach silences TSC and prevents future NaN display.
- **Confidence:** HIGH

### [P2] PLANS-02 — `PlanDetailModal` fetches plan detail via `getPlan()` (raw `api/plans` call) without a tier branch; free users who somehow get a `selectedPlanId` will make a failing server round-trip

- **File:** `mobile/app/(tabs)/plans.tsx:413–422` (`loadPlan` callback), `plans.tsx:1106–1115` (modal rendered when `selectedPlanId !== null`)
- **Problem:** `PlanDetailModal.loadPlan()` calls `getPlan(planId)` unconditionally — there is no `isLocalFirst` / `syncsToServer` branch. In the current UI, `selectedPlanId` is only set via `setSelectedPlanId(p.id)` inside the `proSync ? (...) : null` branch (line 1039–1091), so free users should not be able to set it through the normal plan-list flow. However, `handleGenerate` (line 959–976) also calls `setSelectedPlanId(result.plan_id)` after a server generate — and `handleGenerate` itself is only wired to `onRegenerate` on `PlanDetailModal` (line 1113), not directly exposed in the free-tier UI. The tier gate is therefore "soft" — it relies on the free-tier UI not exposing the trigger, not on an explicit `isLocalFirst` check inside `loadPlan`. If the `handleGeneratePress` path is ever wired to a free-tier control, or `selectedPlanId` is set by any other path, free users will hit a 401/403/500 on `getPlan()`.
- **Evidence:**
  ```ts
  // plans.tsx:413–422
  const loadPlan = React.useCallback(() => {
    setIsLoading(true);
    setError(null);
    getPlan(planId)          // raw api/plans call — no isLocalFirst branch
      .then(setPlan)
      .catch((err) =>
        setError(err instanceof Error ? err.message : 'Failed to load plan')
      )
      .finally(() => setIsLoading(false));
  }, [planId]);
  ```
- **Invariant/Rubric:** Invariant 1 (local-first) — indirect violation via absent explicit guard; P2 (soft gate, not currently reachable, but fragile).
- **Suggested direction:** Add an explicit `if (isLocalFirst(user)) { setError('Server plan detail is a Pro feature'); return; }` guard at the top of `loadPlan`, or conditionally skip rendering `PlanDetailModal` when `isLocalFirst(user)` is true (the `LocalPlanModal` already handles the on-device case). This hardens the gate so it doesn't depend solely on UI wiring.
- **Confidence:** MED (not currently reachable on the free path, but the guard is absent)
