### [P0] A6-01 — Tier-branch violation: `handleStartWorkout` fires `POST /workouts` for free/local-first users

- **File:** `mobile/app/templates.tsx:334–353`
- **Problem:** `isLocalFirst(user)` only gates `fetchTemplates()` (line 275), which correctly skips the GET /templates for free users and shows bundled programs instead. However `handleStartWorkout()` (line 334) calls `apiClient.post('/workouts', …)` unconditionally — no `isLocalFirst` branch. A free user who taps "Start Workout" on a bundled-program detail-day triggers this path and hits a failing/slow authenticated endpoint (Invariant 1). The bundled-program flow routes through a separate modal (`bundledSelected`) whose tap handler navigates with `router.push(…bundledProgram=…)` and never calls `handleStartWorkout`, so in practice only the server-template flow (Pro-only section) can reach this. However the guard is absent at the function level: if the `selected` template state is ever set for a free user (e.g. via deep link) the POST fires.
- **Evidence:**
```ts
const handleStartWorkout = async () => {
  if (!selected || !(selected.sessions?.length)) return;
  setStarting(true);
  try {
    const firstSession = selected.sessions[0];
    await apiClient.post('/workouts', {      // ← no isLocalFirst check
      template_id: selected.id,
      session_name: firstSession.session_name,
      exercises: firstSession.exercises,
    });
```
- **Invariant/Rubric:** Invariant 1 (local-first tiering) — P0 tier-branch violation.
- **Suggested direction:** Add `if (isLocalFirst(user)) { /* write to localDb instead of /workouts */ setSelected(null); router.push('/(tabs)/log' as any); return; }` before the `apiClient.post` call, or at minimum a guard that prevents the POST path from being entered by free users.
- **Confidence:** HIGH

### [P0] A6-02 — `staggerStyle` receives `Animated.Value | undefined` in data-export.tsx and recovery-code.tsx: crash risk if array index is undefined

- **File:** `mobile/app/data-export.tsx:433,460,526,569,666` and `mobile/app/recovery-code.tsx:238,264,299,343`
- **Problem:** `staggerStyle(anim: Animated.Value)` (data-export.tsx:82) takes a non-optional `Animated.Value`. It calls `anim.interpolate(…)` unconditionally. `staggerAnims` is `Animated.Value[]`; indexing it with `[N]` under TypeScript strict mode returns `Animated.Value | undefined` (TSC errors TS2345 confirmed in 00-tsc.txt). If the `count` passed to `useStaggerFade` is ever lower than expected (e.g. re-render with a different count), the call `staggerAnims[N]` returns `undefined` and `anim.interpolate(…)` throws at runtime. In `training-survey.tsx` the same pattern was fixed with a null guard in `staggerStyle`; the `data-export.tsx` copy was not updated. `recovery-code.tsx` has the same unfixed pattern.
- **Evidence (data-export.tsx):**
```ts
function staggerStyle(anim: Animated.Value): object {   // non-optional
  return {
    opacity: anim,
    transform: [{ translateY: anim.interpolate({ … }) }],
  };
}
// …
<Animated.View style={reduceMotion ? undefined : staggerStyle(staggerAnims[0])}>
```
TSC: `app/data-export.tsx(433,71): error TS2345: Argument of type 'Value | undefined' is not assignable to parameter of type 'Value'.`
- **Invariant/Rubric:** P0 correctness — crash on undefined access. Also P1 (null-safety).
- **Suggested direction:** Copy the null guard from `training-survey.tsx:288–291` (`if (!anim) return {};`) into `staggerStyle` in both `data-export.tsx` and `recovery-code.tsx`, or change the parameter type to `Animated.Value | undefined`.
- **Confidence:** HIGH

### [P1] A6-04 — `csv-import.tsx:handleImport` missing `localFirst` in `useCallback` deps

- **File:** `mobile/app/csv-import.tsx:107–144`
- **Problem:** `handleImport` is wrapped in `useCallback` with deps `[file]`. It reads `localFirst` (a const derived from `user` and `isLocalFirst()`) inside the callback but does not include it in the deps array. If the user's tier changes in-session (e.g., after an upgrade from Free to Pro), the closure retains the stale `localFirst = true` value and the user sees the error "CSV import needs a synced (Pro) account" even though they just upgraded. `localFirst` itself is not a state variable (it's computed from `user` on each render), so the fix is to add it to the deps array.
- **Evidence:**
```ts
const handleImport = useCallback(async () => {
  if (!file) return;
  if (localFirst) {    // ← captured at callback creation time
    setUploadError('CSV import needs a synced (Pro) account…');
    return;
  }
  // …
}, [file]);   // ← localFirst missing from deps
```
- **Invariant/Rubric:** P1 — stale closure / incorrect behaviour after tier upgrade.
- **Suggested direction:** Add `localFirst` to the `useCallback` deps array: `}, [file, localFirst]);`
- **Confidence:** HIGH

### [P1] A6-05 — `_layout.tsx`: `isLoading` spinner blocks first render instead of rendering cached user immediately (Invariant 5 partial)

- **File:** `mobile/app/_layout.tsx:170–176`
- **Problem:** `RootNavigator` returns a full-screen `ActivityIndicator` when `isLoading` is `true` (line 170–176). This blocks the entire navigation tree. Invariant 5 requires rendering the cached user immediately. However the AuthContext implementation (AuthContext.tsx:363–371) does restore the cached user from SecureStore and then sets `setIsLoading(false)` before the background refresh (line 371), so the window is only the synchronous SecureStore read duration (~5–50 ms). This is acceptable per the invariant's spirit. However `index.tsx:34–44` also shows a spinner when `isLoading` is true and redirects to `/(auth)/login` when not authenticated, which is correct. The risk is if `isLoading` stays true indefinitely — which is guarded by the `finally` block in AuthContext that always calls `setIsLoading(false)`. No clear bug, but the `finally` at line 441 of AuthContext only fires after the background refresh completes (success or failure including the 8s timeout), meaning the spinner is shown for up to ~8 seconds + SecureStore read on cold start if there IS a stored token and the server is slow. The cached user IS shown to the Stack once `isLoading(false)` fires from line 371, but the `<ActivityIndicator>` in `_layout.tsx:170` still blankets the whole tree during the early phase.
- **Evidence:**
```ts
if (isLoading) {
  return (
    <View style={[styles.loadingContainer, { backgroundColor: theme.colors.bgPrimary }]}>
      <ActivityIndicator size="large" color={theme.colors.accentDefault} />
    </View>
  );
}
```
- **Invariant/Rubric:** Invariant 5 partial concern — spinner blocks until after cached user is restored, then background refresh does NOT re-show the spinner (isLoading goes false at line 371 before the background refresh). Overall pattern is correct; this is MED-confidence concern about the window.
- **Suggested direction:** Confirm the `setIsLoading(false)` at AuthContext.tsx:371 always fires before the background `Promise.race`, not after. The current code does call it before `await Promise.race(…)` so the invariant is satisfied. Flag as LOW risk but worth a code comment clarifying the invariant is upheld.
- **Confidence:** MED

### [P2] A6-08 — `data-export.tsx` cloud-status `useEffect` has no cleanup and silently sets state after unmount

- **File:** `mobile/app/data-export.tsx:186–192`
- **Problem:** The `React.useEffect` that calls `getStatus()` (line 186) uses `.then(setCloudStatus)` — a floating promise with no mounted-check or cleanup flag. If the component unmounts before `getStatus()` resolves (e.g., user navigates away quickly), `setCloudStatus` is called on an unmounted component, triggering a React warning and potentially a stale-state bug if the component is re-mounted before the promise resolves.
- **Evidence:**
```ts
React.useEffect(() => {
  setCloudStatusLoading(true);
  getStatus()
    .then(setCloudStatus)       // ← no mounted check
    .catch(() => {})
    .finally(() => setCloudStatusLoading(false));
}, []);
```
- **Invariant/Rubric:** P2 — missing `useEffect` cleanup / floating promise.
- **Suggested direction:** Add a `let mounted = true;` flag and check it in the `.then` and `.finally` callbacks, returning a cleanup that sets `mounted = false`.
- **Confidence:** HIGH
