### [P0] A5-01 — `insights.tsx`: setState-after-unmount on deload ack (async Alert callback)

- **File:** `mobile/app/insights.tsx:173–191`
- **Problem:** `handleAckDeload` opens an `Alert.alert` with an async `onPress` callback that calls `setDeloadAcking(true)`, awaits `ackDeload()`, then calls `setDeload(...)` and `setDeloadAcking(false)`. There is no mounted-flag guard. If the user navigates away while the POST is in flight (e.g. taps back after confirming the alert), all three `setState` calls fire on an unmounted component. React Native surfaces this as a no-op warning today but it is a latent state-corruption/crash vector, and the `load()` useEffect similarly has no cleanup/cancel flag — rapid navigation in and out re-triggers the three-call `Promise.all` with no cancellation, so the last-to-resolve fetch wins and can overwrite fresher data.
- **Evidence:**
```ts
const handleAckDeload = useCallback(async () => {
  Alert.alert('Start deload week', '…', [
    { text: 'Cancel', style: 'cancel' },
    {
      text: 'Start deload',
      onPress: async () => {
        setDeloadAcking(true);          // ← no mounted check
        const ok = await ackDeload();   // ← can complete after unmount
        if (ok) {
          setDeload((prev) => (prev ? { ...prev, recommended: false } : prev));
        }
        setDeloadAcking(false);         // ← setState after unmount
      },
    },
  ]);
}, []);
```
- **Invariant/Rubric:** P0 — async race condition: `setState` after unmount; `useEffect` with no cleanup/cancel flag.
- **Suggested direction:** Add a `useRef<boolean>(true)` mounted flag, set it to `false` in the `useEffect` cleanup, and gate all `setState` calls in the Alert callback and `load()` finalizer behind `if (mountedRef.current)`. Alternatively use an `AbortController` signal and wrap `load()` in a cancellable pattern.
- **Confidence:** HIGH

### [P1] A5-02 — `groups.tsx`: `SafeAreaView` inside `<Modal>` (Invariant 3 × 2 modals)

- **File:** `mobile/app/groups.tsx:277, 392`
- **Problem:** Both `CreateGroupModal` and `JoinGroupModal` wrap their content in `<SafeAreaView edges={['top','bottom']}>` placed directly inside a React Native `<Modal>`. Per Invariant 3, `SafeAreaView` does not reliably propagate inside an RN `<Modal>` on iPhone — the header row will sit under the Dynamic Island on device. The correct fix is to read `useSafeAreaInsets()` and apply `paddingTop: Math.max(insets.top, 12)` directly to the header `View`.
- **Evidence:**
```tsx
<Modal visible={visible} animationType="none" presentationStyle="pageSheet">
  <Animated.View style={[{ flex: 1 }, sheetAnimStyle]}>
   <SafeAreaView style={[styles.modalContainer, { backgroundColor: theme.colors.bgPrimary }]}
                 edges={['top', 'bottom']}>   {/* ← Invariant 3 violation */}
    <KeyboardAvoidingView style={styles.modalContainer} …>
      <View style={[styles.modalHandle, …]} />
      <View style={[styles.modalHeader, …]}>   {/* ← header not inset-padded */}
```
- **Invariant/Rubric:** P1 — Invariant 3: `SafeAreaView` inside `<Modal>` does not push content below Dynamic Island.
- **Suggested direction:** Replace `SafeAreaView` with a plain `View` in both modals. Import `useSafeAreaInsets` and apply `paddingTop: Math.max(insets.top, 12)` to the `styles.modalHandle` wrapper or `modalHeader` view directly.
- **Confidence:** HIGH

### [P1] A5-05 — `insights.tsx`: array subscript `staggerAnims[n]` yields `Animated.Value | undefined` (4 × TS2345, confirmed in tsc output)

- **File:** `mobile/app/insights.tsx:259, 314, 322, 337`
- **Problem:** `staggerAnims` is typed as `Animated.Value[]` (returned from `useRef(...).current`). TypeScript's `noUncheckedIndexedAccess` (or standard strict array indexing) flags `staggerAnims[0..3]` as `Animated.Value | undefined`, and `staggerStyle()` accepts `Animated.Value` — not `| undefined`. This is confirmed in `audits/full-review-2026-06-19/00-tsc.txt` lines 43–50 as four TS2345 errors. At runtime the array is always length-4 (created with `Array.from({ length: count }, ...)` where `count = 4`), so there is no actual crash risk, but it is a real type gap that would mask a genuine crash if `count` were ever reduced below 4 without updating the call sites.
- **Evidence:**
```ts
// useStaggerFade returns Animated.Value[] (not readonly tuple)
const staggerAnims = useStaggerFade(4, !reduceMotion);
// …
<Animated.View style={reduceMotion ? undefined : staggerStyle(staggerAnims[0])}>
//                                                                          ^^^^ Value | undefined
```
- **Invariant/Rubric:** P1 — confirmed tsc TS2345 null-safety error (not a router-string false-positive).
- **Suggested direction:** Either change `useStaggerFade` to return a fixed-length tuple type (`[Animated.Value, Animated.Value, Animated.Value, Animated.Value]`) or add a non-null assertion at each call site with a comment. A tuple return is cleaner and self-documenting.
- **Confidence:** HIGH

### [P2] A5-06 — `health-metrics.tsx`: `formatDate` destructures `month` without undefined guard (TS18048, confirmed)

- **File:** `mobile/app/health-metrics.tsx:49–50`
- **Problem:** `const [year, month, day] = dateStr.split('-').map(Number)` — array destructuring from a `.split()` result typed as `string[]` means `month` and `day` are `number | undefined` in strict TS. The call `new Date(year, month - 1, day)` at line 50 then passes a potentially-undefined operand, confirmed as TS18048 + TS2345 in `00-tsc.txt`. If `dateStr` is ever malformed (e.g. an unexpected server value), `month - 1` produces `NaN` and the `Date` constructor yields `Invalid Date`, which `toLocaleDateString` renders as `"Invalid Date"` — visible glitch in the day list.
- **Evidence:**
```ts
function formatDate(dateStr: string): string {
  const [year, month, day] = dateStr.split('-').map(Number);
  const d = new Date(year, month - 1, day);  // month possibly undefined → NaN
```
- **Invariant/Rubric:** P2 — TS18048/TS2345 confirmed; "Invalid Date" glitch on malformed input.
- **Suggested direction:** Either guard with `if (!year || !month || !day) return dateStr;` before the `new Date(...)` call, or use the ISO parse directly: `new Date(dateStr + 'T00:00:00')` (avoids timezone shift for date-only strings by using local midnight).
- **Confidence:** HIGH

### [P2] A5-08 — `insights.tsx`: `load()` useEffect has no cleanup — concurrent navigations race

- **File:** `mobile/app/insights.tsx:162–165`
- **Problem:** The `useEffect` that triggers `load()` has no cleanup/cancel return. If the user navigates away and back quickly while the `Promise.all([getReadiness(), getRecovery(), getDeload()])` is in flight, the old and new fetches both resolve and call `setReadiness`/`setRecovery`/`setDeload` — the last to settle wins, not the most recent request. For Pro users on a slow connection this can result in stale server data overwriting fresher data. (Related to A5-01 but distinct: this is a stale-fetch overwrite, not setState-after-unmount per se.)
- **Evidence:**
```ts
useEffect(() => {
  setLoading(true);
  load().finally(() => setLoading(false));
  // ← no cleanup: no cancel flag, no AbortController, no ignore variable
}, [load]);
```
- **Invariant/Rubric:** P2 — missing `useEffect` cleanup; stale fetch can overwrite fresh data on rapid navigation.
- **Suggested direction:** Add an `ignore` flag: `let ignore = false;` inside the effect; check `if (!ignore)` before each `setState` call in `load()`, and return `() => { ignore = true; }` as the cleanup.
- **Confidence:** HIGH

### [P3] A5-09 — `group-detail.tsx`: `getInitial` string-index `[0]` confirmed TS2532

- **File:** `mobile/app/group-detail.tsx:78` (as reported in `00-tsc.txt` line 36)
- **Problem:** `return name.trim()[0].toUpperCase()` — string subscript indexing returns `string | undefined` in strict TS (confirmed TS2532). At runtime `.trim()` on a non-empty string is always safe because the caller already guards `if (!name) return '?'`, but the type gap is real and could mask a regression if the guard is removed.
- **Evidence:**
```ts
function getInitial(name: string | null): string {
  if (!name) return '?';
  return name.trim()[0].toUpperCase();  // TS2532: Object is possibly 'undefined'
}
```
- **Invariant/Rubric:** P3 — confirmed TS2532; logic is safe at runtime but type is unsound.
- **Suggested direction:** Use `name.trim().charAt(0).toUpperCase()` — `charAt` always returns `string` (empty string for out-of-bounds), eliminating the type error.
- **Confidence:** HIGH

### [P3] A5-10 — `groups.tsx`: `CreateGroupModal` / `JoinGroupModal` silently drop the `goal` argument

- **File:** `mobile/app/groups.tsx:488–498, 501–512`
- **Problem:** Both modal `onSubmit` handlers accept `(_goal: WeeklyGoal)` but immediately discard the argument (prefixed `_`). This is a UX mismatch: the UI invites the user to pick a goal, implies it is stored, but silently ignores it.
```ts
const handleCreateSubmit = useCallback(
  async (name: string, _goal: WeeklyGoal) => {   // ← goal discarded

      await createGroup({ name, sizeCap: 12 });   // ← no goal sent

    } catch { /* error shown inside modal */ }
  }, [createGroup]

```
- **Invariant/Rubric:** P3 — misleading UX / dead UI; not a data-integrity bug but a product gap worth flagging.
- **Suggested direction:** Either wire the goal through a subsequent `PUT /goals/weekly` call after the group is created/joined, or remove the `GoalPicker` from these modals entirely and replace with a note that the goal is set in profile settings.
- **Confidence:** HIGH
