Safe-area header

P1

Wrap the logger in a safe-area container so the close control and progress bar sit below the status bar and Dynamic Island on modern iPhones.

The problem: the screen is a full-screen modal whose header starts at y≈12pt — the close chevron lands inside the Dynamic Island zone where iOS intercepts taps, so users can get stuck mid-session. The change: apply SafeAreaView with edges top+bottom; offset the header row by the reported inset value.

Addresses: finding stepper-logger-no-top-inset · Effort: S · Risk: low

56pt steppers + decimal keypad

P2

Enlarge the weight and reps stepper buttons to a minimum 56pt touch target and switch the keyboard to the decimal type so fractional weights (82.5 kg) can be entered without extra taps.

The problem: the current −/+ controls are ~22pt, well below Apple's 44pt minimum, causing frequent mis-taps. The keyboard is a plain number pad which lacks a decimal point, so users can't input fractional plates without a workaround. The change: resize buttons to 56×56pt, set inputMode="decimal" on the weight field, and apply a 2.5 kg / 1 rep increment on hold.

Effort: S · Risk: low · Ships best with: option 3 (tabular display) and option 10 (unit pref)

Large tabular set numbers

P2

Render the current weight × reps value at heading size (≥24pt) using font-variant-numeric: tabular-nums so the readout is legible at a glance and does not jitter horizontally as digits change.

The problem: the active set display is small (~14pt) and uses proportional figures, so incrementing from 9→10 reps shifts the layout. Athletes glance at this screen between sets in loud, dim gyms — the number must be instantly readable. The change: promote the weight×reps to a prominent, centred display using tabular numerals; keep secondary metadata (set count, RPE) in the subtitle row.

Effort: S · Risk: low · Ships best with: option 2 (bigger steppers) and option 10 (unit pref)

Inline rest-timer ring

NEW

Auto-start a rest countdown inside the logger immediately after the user taps "Log set". A circular progress ring counts down the rest period; tapping anywhere on it skips to the next set.

The opportunity: useRestTimer already exists in the codebase but is never rendered anywhere visible. Athletes need a rest cue between sets — without one they either wait too long (cold) or rush (underrecovered). The change: show a ring + countdown row after each logged set; allow skip; ring auto-hides when time elapses. Rest duration comes from the plan's rest_seconds value, defaulting to 90 s.

Effort: M · Risk: low (hook already exists) · Dependency: none

"Last: 80×5" ghost + copy

NEW

Show a faded reference line above the weight stepper displaying the weight and reps logged for this exercise in the most recent previous session. One tap copies both values into the fields.

The opportunity: progressive overload requires remembering last week's numbers. Currently users rely on memory or leave the app to check history. P2

Slim animated progress bar

P2

Replace the dot-based exercise indicator with a 4pt determinate progress bar (exercise i of n) that animates on exercise change and an "i / n" fraction label in the header.

The problem: dot indicators overflow and become unreadable for sessions with seven or more exercises, which is a common plan structure. The fraction label "2/6" is already present but small. The change: remove the dots; render a full-width teal bar below the header using width: (current/total * 100)% with a 300ms ease animation; keep the "i / n" fraction in the header trailing slot.

Effort: S · Risk: low · Dependency: none

Swipe between exercises

NEW

Wrap the exercise content in a horizontal pager so users can swipe left/right to move between exercises within the session, while keeping the existing prev/next buttons as visible fallback controls.

The opportunity: swipe navigation is the muscle-memory interaction model for adjacent content on mobile. The current button-only approach forces two-handed use or precise tapping while holding a barbell. The change: use FlatList in horizontal paginated mode or PagerView; keep the ‹/› chevron buttons flanking the content as explicit affordances; animate pager dots (max 5 shown, scrolling). The prev/next buttons remain for accessibility.

Effort: M · Risk: medium (gesture conflict with scroll) · Dependency: none

RIR/RPE chip row

P2

Replace the free-text effort field with a horizontal chip row of four options — Easy / 2 RIR / 1 RIR / Fail — that maps to RPE 6/8/9/10. One tap selects; tapping again deselects (optional field).

The problem: a free numeric entry for RPE adds keyboard friction and most users skip it entirely, meaning effort data is absent from the history view and the strength model. A tap-select chip row has near-zero cost for the user. The change: render four chips below the set controls; map to rpe values 6, 8, 9, 10; persist with the set row. Label copy can be adjusted — "2 RIR" and "1 RIR" are meaningful to intermediate lifters but could be simplified to "moderate / hard / very hard / maxed out" for broader audiences (founder's call).

Effort: S · Risk: low · Dependency: none (adds new optional column)

Unit-aware weights

P1

Read the user's unit_pref (kg / lb) from profile and display all weights in the logger with the correct unit suffix. A compact kg|lb toggle in the weight card lets the user flip on the fly.

The problem: the logger assumes kg throughout. US-based users logging in pounds see their input treated as kilograms downstream — the strength model, percentiles, and history graphs all become wrong. The change: pull unit_pref from local profile; multiply/divide when reading and writing set rows; show a "KG" or "LB" label beneath the big number; add a compact segmented toggle. The increment step should also respect units (2.5 kg ≈ 5 lb).

Addresses: finding units · Effort: S · Risk: medium (data migration for existing lb users) · Ships best with: options 2 and 3

Log confirmation micro-animation + haptic

P2

After a successful set save, briefly animate the "Log set" button (scale pulse + checkmark swap, 150 ms) and trigger a light haptic to give clear physical confirmation that the set was recorded.

The problem: the current UI gives no feedback after tapping "Log set" — the button stays static. Users frequently double-tap because they can't tell if the save worked, creating duplicate set entries. The change: on save success, run a Animated.sequence scale pulse (1 → 1.06 → 1) and swap the label to "✓ Set logged!" for 800 ms; call Haptics.impactAsync(ImpactFeedbackStyle.Light). On save failure, skip the animation and show option 14 (retry toast) instead.

Effort: S · Risk: low · Dependency: expo-haptics (already in package.json)

12

Sticky bottom action bar

P1

Pin "Log set" (primary) and "Finish workout" (secondary) in a fixed action bar at the bottom of the screen, above the home indicator inset, so they are always reachable regardless of scroll position.

The problem: the action buttons scroll with the page content. On devices with longer content (notes field, RIR chips, previous-set history rows) the primary action disappears below the fold — or worse, sits directly over the home indicator where iOS intercepts taps. This is especially bad mid-set when a user has gloves on. The change: extract the two action buttons into an absolutely-positioned (or KeyboardAvoidingView-aware) bar pinned to the bottom safe area; the scrollable content stops at the bar's top edge.

Effort: S · Risk: low · Dependency: pairs well with option 1 (top inset)

"Add exercise" empty state

P0

Replace the blank "No exercises in this session" text with a purposeful empty state that includes an icon, a one-sentence explainer, and a prominent "+ Add exercise" CTA button.

The problem: when a session loads with zero exercises (triggered by a data-side template-unwrap mismatch, now fixed, but also possible for free-form sessions) the user sees a blank sheet with dead-end text and no path forward. The change: detect exercises.length === 0 and render an empty-state panel instead of the logger stepper. The CTA navigates to the exercise-library picker. This is a P0 because users are left stuck with no recovery action — even after the data fix, free-form sessions legitimately start empty.

Addresses: finding gettemplate-unwrap-mismatch (UX side) · Effort: S · Risk: low

14

Save-failure retry toast

P2

When a set save fails (network error, DB write error), show a non-blocking toast at the bottom of the screen with "Couldn't save set" and a tappable "Retry" action. Auto-dismiss after 6 seconds.

The problem: the current error path silently drops the set — the user has no indication the data was lost and no way to recover without re-entering values. In a local-first architecture a write failure is rare but possible (sync conflict, storage full). The change: catch the rejected promise from the set-save call; hold the failed set payload in state; render a toast overlay (not a blocking modal); the Retry button replays the save call with the preserved payload. Dismiss on success or after 6 s.

Effort: S · Risk: low · Dependency: pairs with option 11 (success animation)

★

Recommended build order

Core six (ship first): options 1, 4, 7, 11, 12, 13 form the highest-impact, lowest-risk batch. Together they fix the three known blocking issues (safe area, sticky actions, empty state), add the most-requested missing feature (rest timer), and make the screen feel complete with a progress bar and set confirmation. None has external dependencies; none risks data integrity. This batch can ship in a single sprint.

Numbers + units pass (second batch): options 2, 3, and 10 should ship together because they each touch the weight display and input layer. Doing them as a combined pass avoids touching those components three separate times and ensures the decimal keyboard, tabular display, and unit conversion all behave consistently.

Remainder: options 5 (last-session ghost), 6 (plate-math icon), 8 (swipe navigation), 9 (RIR chips), and 14 (retry toast) are individually small — each is an S-effort item — and can be tackled opportunistically or batched as a "polish sprint" after the core six land.

Option 8 (swipe) carries the only medium-risk flag due to gesture conflicts with vertical scroll; consider a brief spike to validate the gesture coexistence before committing it to the sprint.
