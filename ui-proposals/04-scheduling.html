1

Safe-area header

P1

The sheet header (✕ · "Create schedule" · Save) renders behind the iOS status bar and Dynamic Island. On iPhone 14 Pro and later the close button and title are completely obscured.

The problem: ScheduleEditorSheet opens inside a bottom sheet that does not propagate SafeAreaView insets to its internal header — the modal container starts at y=0 of the screen, so the header rows sit directly under the status bar.  The change: Wrap the sheet header row with SafeAreaView edges={['top']} (or apply the top inset value via useSafeAreaInsets() as a paddingTop on the header container). No layout changes needed below the header.

Addresses: schedule-editor-header-under-status-bar  ·  Effort: S (1–2 lines)  ·  Risk: Low

Kill the infinite spinner

P0

Opening the schedule editor produces a spinner that never resolves. The sheet body remains blank indefinitely — the form is completely unusable.

The problem: The sheet's useEffect calls localDb.getSchedules() on mount. That call acquires an exclusive write lock on the SQLite connection. A concurrent migration check (triggered by the same mount cycle) holds an identical lock, so both tasks wait on each other — a classic self-deadlock. Neither the read nor the migration ever completes, and the loading state never resolves.  The change: Serialise migration and query startup through a single boot promise so the migration always finishes before query hooks are allowed to run. The fix is already merged to localDb.ts; this option tracks the UI-side confirmation that the spinner is gone.

Addresses: localdb-migration-snapshot-deadlock  ·  Effort: S (fix already in localDb.ts)  ·  Risk: Low

Segmented control with animated thumb

P2

The "Day of week / Repeating cycle" toggle is rendered as two plain tappable text labels with no visual distinction between selected and unselected states. There is no animated transition, making it hard to confirm a selection happened.

The problem: The current implementation uses two sibling Pressable elements with inline style changes on press — no shared background, no thumb animation, and hit targets that depend on font size.  The change: Replace with a properly styled segmented control: a pill-shaped outer track, a sliding white/teal thumb that animates between segments via Animated.spring, and each segment label rendered at a fixed height with a comfortable tap target. When "Repeating cycle" is selected the day-of-week picker fades out and the cycle stepper (option 5) fades in.

4

Day-of-week chips

P2

The existing day-of-week picker uses small text labels in a horizontal list. Tap targets are too narrow for one-handed use, and there is no feedback summary showing which days are currently active.

The problem: Each day label is a Text with a Pressable wrapper sized to the character width — approximately 16px — well under Apple's 44pt minimum recommended touch target. On small screens adjacent days are easy to mis-tap.  The change: Replace with 7 circular chips, each 36×36dp, that toggle on tap with a teal fill animation. Below the chips show a live summary count ("4 days selected") and a row of selected-day name chips. The circular layout matches the calendar metaphor users already know from native iOS.

Addresses: schedule-editor-day-tap-targets  ·  Effort: S  ·  Risk: Low

5

Repeating-cycle stepper

NEW

When "Repeating cycle" is selected in the frequency toggle, there is currently no control to configure the cycle length. The mode exists in the data model but has no usable UI.

The problem: The repeating cycle path in ScheduleEditorSheet renders an empty container — the mode is effectively dead in the UI even though the schema supports it.  The change: Show two steppers: "Train days" (1–7) and "Rest days" (1–4), defaulting to 3 train / 1 rest. Below the steppers render a text-based cycle preview ("Train · Train · Train · Rest · …") so the user sees the pattern before saving. The cycle is stored as {trainDays: number, restDays: number} on the schedule record.

Addresses: repeating-cycle-no-ui  ·  Effort: M  ·  Risk: Low

6

Routine-per-slot picker

NEW

The schedule builder lets you choose which days to train, but there is no way to say what to train on each day. A "Mon / Wed / Fri" schedule is meaningless without specifying the workout for each slot.

The problem: Schedule records store selected days but have no routine_id field per slot. The user has to mentally cross-reference their schedule with their routine list every session.  The change: After days are selected, show an "Assign routines" section — one row per active day. Each row has the day name and a picker chip that opens a bottom sheet of the user's saved routines. "Rest" is always available as a first-class option (rendered in a muted red chip). This requires a schedule_slots join table (schedule_id, day_index, routine_id nullable).

Addresses: schedule-no-routine-assignment  ·  Effort: M  ·  Risk: Med (schema change)

Live 2-week preview

NEW

There is no preview of what a schedule will look like in practice. The user has to save and navigate to the calendar before they can see the pattern they have built.

The problem: The editor is a pure input form with no output feedback. Users cannot see if their selected days result in back-to-back training days or awkward gaps until after saving.  The change: Below the day picker render a live 2-week mini-grid that redraws as days are toggled. Each training day is filled teal; rest days are muted. A summary line ("8 sessions in the next 14 days") gives an at-a-glance volume check. For repeating cycle mode the preview projects the cycle forward from today. Pure derived state — no network calls.

8

Sticky Save with validation reason

P2

The Save button lives in the navigation header and is trivially easy to miss. When the form is invalid Save is grayed out but there is no explanation of what is missing, leaving the user stuck.

The problem: A header-bar Save is common in older iOS patterns but performs poorly on large phones — it is physically far from the content and hard to reach one-handed. The silent disabled state gives users no recovery path.  The change: Add a sticky footer bar that pins to the bottom safe area (above the home indicator). The bar contains a full-width primary CTA ("Save schedule") that is disabled (muted teal) when the form is invalid, with a short comma-separated reason line below it ("Name required · pick at least 1 day"). The header Save remains as a secondary shortcut for familiarity. On valid forms the CTA becomes fully teal.

Addresses: schedule-editor-save-discoverability  ·  Effort: S  ·  Risk: Low

9

Time-of-day and push reminder

NEW

Schedules have no concept of time. There is no way to set when you plan to train, and no push notification reminder tied to the schedule. Users rely on calendar alerts outside the app.

The problem: A schedule without a time is an intention without a trigger. The absence of in-app reminders is a primary reason users abandon habit loops within the first two weeks.  The change: Add an optional "Workout time" row (tapping opens a wheel time picker, stored on the schedule record). Below it add a "Reminder" toggle (off by default). When toggled on, the user is prompted for notification permission and a 15 min before local notification is scheduled for each active training day using expo-notifications. A "Next reminder" preview row updates live. The offset (15 min) is hardcoded in v1; a stepper for offset can follow in a later iteration.

Addresses: schedule-no-reminder  ·  Effort: M  ·  Risk: Med (notification permissions flow)

10

Confirm-on-dismiss

P2

Tapping ✕ (or swiping down on the sheet) immediately dismisses the editor and silently discards any unsaved changes. There is no warning or recovery path.

The problem: The bottom sheet uses onDismiss with no guard. Even after the user has typed a name, selected days, and assigned routines, a stray downward swipe destroys all input without feedback.  The change: Track a isDirty boolean (set to true as soon as any field changes from its initial value). If isDirty is true when ✕ is pressed or the sheet is dragged to dismiss, intercept the gesture and show a modal action sheet ("Discard changes? / Keep editing · Discard"). Only "Discard" proceeds with dismissal. If the form is untouched, dismiss immediately as before — no friction for a clean exit.

11

Slide-in sheet motion

P2

The schedule editor sheet appears instantly — it cuts onto screen with no entrance transition. The content origin is ambiguous: is it a new screen, a modal, or a panel layered over the current page?

The problem: The sheet uses the default BottomSheetModal with animateOnMount={false} (set to avoid a jank at open time). This eliminates the animation that communicates the spatial relationship between the trigger and the editor.  The change: Re-enable animateOnMount and configure a spring-based entrance: the sheet slides up from below the trigger point and fades in simultaneously (opacity: 0→1, translateY: 60→0, spring tension 120 / friction 20). The underlying plans screen scales down very slightly (0.97→1.0) to reinforce depth. Closing reverses the animation. Total duration ≈ 220ms — fast enough to feel snappy, slow enough to convey origin.

Addresses: schedule-editor-no-entrance-animation  ·  Effort: S  ·  Risk: Low

12

First-time explainer banner

NEW

New users who navigate to the schedule editor do not know what a "schedule" is in Peak Fettle's model or how it differs from a routine or a plan. The editor opens directly onto an empty form with no context.

The problem: The term "schedule" overlaps with several related concepts (plan, program, routine) in fitness-app vocabulary. Without explanation, new users frequently create a schedule with no routines assigned (see option 6) or mistake it for a one-off workout entry.  The change: On the very first open (tracked via AsyncStorage key seen_schedule_explainer), render a teal-tinted info banner at the top of the sheet body: a calendar emoji, a bolded "What is a schedule?" heading, and a single plain-English sentence explaining the concept. A small ✕ on the banner dismisses it permanently. Shown once — never again after the user acknowledges it.

Addresses: schedule-editor-no-onboarding-context  ·  Effort: S  ·  Risk: Low

13

Inline validation messages

P2

When the form is invalid, Save is silently disabled. There is no indication of which fields are incomplete or what the user needs to do to unblock the button. The form appears to be frozen.

The problem: Validation runs in a useMemo but the result is only used to disable the Save button — the individual field errors are never surfaced.  The change: After the first attempted tap on the disabled Save (tracked with a submitAttempted boolean), show inline error messages directly beneath each invalid field: a red border on the text input, and a small errtx-styled line underneath ("Schedule name is required" / "Pick at least one training day"). Errors clear as soon as the field becomes valid. This is the standard iOS form validation pattern — users know exactly what to fix without hunting.

14

Local-first schedule persistence

P1

Schedules currently save to Supabase only. Unauthenticated (free) users cannot create or persist a schedule at all — the save call fails silently if there is no active session. There is no offline fallback.

The problem: The schedule save path calls supabase.from('schedules').upsert() directly. If the user is not signed in — or is signed in but offline — the call fails and the error is swallowed. The schedule is lost. This is especially painful as schedule creation is one of the first things a new user does before they have connected an account.  The change: Write schedule records to the local SQLite database first (via localDb) and show a "Saved on-device" confirmation pill immediately. If the user is signed in and online, background-sync the record to Supabase using the existing PowerSync layer and update the pill to "Synced" when the round-trip completes. Free / offline users see "Works offline · syncs when signed in." This aligns with the local-first architecture described in TICKET-094.

Addresses: local-first-schedule-persistence (TICKET-094)  ·  Effort: S–M  ·  Risk: Low (additive write path)

Recommendation

What to build and in what order

These 14 options split cleanly into three tiers. Ship the first tier now — it fixes the screenshot bugs you already know about. The second tier turns the editor into something genuinely usable in a single sprint. The third tier is polish and power features you can sequence by feel.

Tier 1 — must ship (fix the screenshot bugs):

Option 1 (safe-area header) and option 2 (kill the spinner) are not optional — they are the two bugs visible in your screenshot. The header is obscured on every modern iPhone; the spinner means the form is literally unusable. Both are one-to-two line fixes. Ship these together in a single commit before anything else.

Tier 2 — usable builder (one sprint, ~3 days):

Option 3 (segmented control) and option 4 (day chips) make the core interaction feel native and correct — low effort, high visible impact. Option 6 (routine per slot) is the most meaningful feature addition: a schedule without routine assignment is a list of days, not a training plan. Option 8 (sticky Save) fixes discoverability and replaces the silent-disable with a helpful reason. Together these four complete the MVP of the screen.

Tier 3 — polish and power (next two sprints):

Option 5 (repeating cycle stepper) unblocks a mode that is currently dead UI. Option 7 (2-week preview) and option 9 (time + reminder) are the features most likely to drive daily active use — the preview reduces scheduling mistakes; the reminder closes the intention-action loop that most training apps fail at. Option 10 (confirm on dismiss) and option 13 (inline validation) are small, low-risk quality closes. Option 11 (slide-in motion) and option 12 (first-time explainer) are one-day additions with outsized polish returns. Option 14 (local-first persistence) is the right architecture call and should be sequenced when the broader TICKET-094 local-first migration lands — it is already the stated direction.
