## P0

### `set-number-column-missing` — set_number column referenced in two export queries does not exist on `sets` table
- **bug** · `peak-fettle-agents/server/routes/user.js:108` · confidence: high
- The `sets` table only has `set_index` (see sets.js lines 91–97 and the 20260612_ensure_schema_complete migration). `set_number` is never defined in any migration. Both GET /user/data-export and GET /user/export SELECT `s.set_number` from the sets table; Postgres will throw error 42703 (undefined_column) on every call, returning a 500 to the user. The GDPR data export endpoint — required to be functional for legal compliance — is permanently broken.
- **Fix:** Change `s.set_number` to `s.set_index` on lines 108 and 641 of user.js. Also verify the column name against the canonical schema migration before deploying.

### `free-user-server-calls` — Free-tier users make personal REST calls to the server, violating local-first invariant
- **local-first** · `peak-fettle-agents/server/routes/workouts.js:52` · confidence: high
- Every personal-data route (workouts, sets, routines, health-metrics, constraints, plans, percentile, insights, csvImport, backup) is accessible to free-tier users because the server has no enforcement of the local-first invariant. The architecture document says 'Free users = on-device SQLite only, MUST make NO personal REST calls', but the server applies only requireAuth (JWT check) with no tier gate. This means free-user training data is still flowing to and stored on the server, contrary to the decision in 20260612_purge_free_user_training_data.sql and the founder decision. Note: /plans/generate and /exercises/:id/alternatives do enforce a paid gate, but all raw data-write routes do not.
- **Fix:** Enforce the local-first boundary at the server level. For routes that store personal training data (POST /workouts, POST /sets, POST /routines, POST /health-metrics, POST /constraints, etc.), add a paid-tier middleware check (as is done in plans/generate). Alternatively, mount a shared tierPolicy middleware at the router level for these routers. The purge migration already exists — apply it once the gate is in place.

### `restore-no-transaction` — restoreBackupToDb does DELETE+INSERT per table outside any transaction — data loss on crash mid-restore
- **data-integrity** · `mobile/src/data/backup/exportEngine.ts:213` · confidence: high
- restoreBackupToDb iterates over 21 tables, issuing a DELETE followed by INSERT rows for each. There is no enclosing transaction. If the device loses power, the app is killed, or any single INSERT throws (e.g. a constraint violation on a malformed backup row), the restore is left half-complete: some tables are wiped and empty while others still have the old data. The user ends up with a corrupt, partially-restored database with no way to recover except re-triggering a restore. For personal health and training data, partial deletion is functionally data loss.
- **Fix:** Wrap the entire restore loop in a SQLite transaction: execute `BEGIN` before the loop and `COMMIT` after. On any thrown error, execute `ROLLBACK`. This ensures the restore is atomic — either all tables are replaced or none are. The MinimalDb interface already supports execute() so `await db.execute('BEGIN')` / `await db.execute('COMMIT')` are valid.

### `streak-col-mismatch` — GET /user/data-export queries nonexistent streaks columns → 500 for all users
- **bug** · `peak-fettle-agents/server/routes/user.js:137` · confidence: high
- The streaks table (db/schema.sql line 197) has columns current_streak_days, longest_streak_days, and last_session_date. The GET /user/data-export query aliases them as current_streak, longest_streak, last_activity_date — none of which exist. Postgres rejects this at query-plan time with error 42703, making every call to GET /user/data-export throw an unhandled 500. This is a GDPR/user-rights endpoint so the failure affects all authenticated users.
- **Fix:** Change the SELECT to: `SELECT current_streak_days, longest_streak_days, last_session_date FROM streaks WHERE user_id = $1`

### `free-rest-day-server` — Free users hit REST server for logRestDay / undoRestDay
- **local-first** · `mobile/app/(tabs)/index.tsx:473` · confidence: high

### `stale-closure-rest` — handleStepperLogSet has stale closure over restDefault and restTimer
- **bug** · `mobile/src/components/WorkoutLoggerHost.tsx:608` · confidence: high
- The callback captures `restDefault` (a state value updated by `cycleRestDefault`) and `restTimer` (the hook object) at the time the callback is created. Because neither appears in the dep array, the callback is never recreated when the user cycles the rest preset. After the user changes the rest default (e.g. 90 s → 120 s), subsequent sets still schedule the old value. Concretely, `setRestSecondsLeft(restDefault)` and `restTimer.start(restDefault)` on lines 601–603 both use the stale captured value from the first render. ### `edit-set-weight-unit-bug` — Edit-set path in StepperLogger populates inputs with raw display string from LoggedSet, not a proper kg round-trip — potential data-integrity on re-save
- **data-integrity** · `mobile/src/components/StepperLogger.tsx:792` · confidence: high
- When a user taps a logged set chip to correct it, `handleEditChip` fills `weight` with `s.weight`, which is the raw user-typed string (e.g. '185' for a lbs user). When the correction is submitted via `handleLogSet`, that string is passed verbatim to `onUpdateSet` (StepperLogger line 752), which calls `handleStepperUpdateSet` in WorkoutLoggerHost. However, there is a second subtlety: the `lastPayloadRef` (line 552 in StepperLogger) is NOT updated on the edit path (line 759: `return` before `lastPayloadRef.current = ...`), so the retry-save button retries the PREVIOUS new-set payload, not the edit. - **Fix:** Update `lastPayloadRef.current` before the early `return` on the edit path (after line 756), or set it to null on the edit path so the retry button is hidden/disabled during edits. Also ensure the edit path sets `setRetryToast(false)` if an earlier toast was showing.

### `ghostlast-double-convert` — ghostLast converts lastTopSetDisplay.weight a second time through displayToKg+kgToInputValue — double-conversion corrupts the prefill value
- **data-integrity** · `mobile/src/components/StepperLogger.tsx:664` · confidence: high
- WorkoutLoggerHost computes `lastTopSetDisplay.weight` as a display-unit value (already converted from kg to lbs when `unitPref === 'lbs'`). StepperLogger then calls `displayToKg(w, unitPref)` which treats that already-converted lbs value as lbs and converts it BACK to kg, then `kgToInputValue` converts to display again. For kg users the double conversion is identity, so no bug. For lbs users: a 185 lb last set becomes `displayToKg(185, 'lbs') = 83.91 kg`, then `kgToInputValue(83.91, 'lbs') ≈ 185.0` — which appears correct because the round-trip is stable. However, `copyLastSet` fills the stepper input with this string; the stepper then sends it to `handleStepperLogSet` which calls `displayToKg` AGAIN (WLH line 545). For a 185 lb value that correctly round-trips, but for non-standard weights (e.g. 187.5 lb → 85.05 kg → 187.58 lb after round-trip) there is floating-point accumulation. The fix is to not double-convert: `ghostLast` should use `kgToInputValue` directly on the stored-kg value, not on a pre-converted display value.
- **Fix:** In WorkoutLoggerHost, pass `lastTopSetDisplay` in stored kg (same as `ls.weight_kg`), not pre-converted display units. Let StepperLogger's `ghostLast` memo call `kgToInputValue(w, unitPref)` once where `w` is the raw kg. Alternatively, keep the WLH contract as-is (display units) but remove the `displayToKg` in StepperLogger line 664 and just call `kgToInputValue(displayToKg(w, unitPref), unitPref)` only if the prop is documented as kg — currently both sides disagree silently.

### `handleexerciseselect-offbyone` — handleExerciseSelect sets currentIndex to exercises.length (out-of-bounds) when adding to existing free session
- **bug** · `mobile/src/components/WorkoutLoggerHost.tsx:499` · confidence: high
- When the user adds a next exercise in a free session (`freeSessionPickerMode=true` and `isExistingSession=true`), `handleStepperAddOffRoutine` appends the new exercise at the end (setting `exercises.length` before append + 1). Then `currentIndex` is set to the original `prev.exercises.length` which, after the append, points one past the end of the new array — i.e., still out of bounds by one. `exercises[currentIndex]` in StepperLogger will be `undefined`, triggering the empty-state render instead of showing the newly added exercise.
- **Fix:** After calling `handleStepperAddOffRoutine`, compute the correct new index: the new exercise was appended at `prev.exercises.length` (before insertion), so `currentIndex` should be set to `prev.exercises.length` (which equals the new last index after insertion). The `setRoutineSession` update inside `handleExerciseSelect` should be `return { ...prev, currentIndex: prev.exercises.length }` where `prev` is the state BEFORE the addition — but since `handleStepperAddOffRoutine` runs a separate `setRoutineSession` call, the two state updates may race. The safer fix is to combine both mutations into a single `setRoutineSession` call that appends and sets the correct index atomically.

### `WL-003` — Ghost/copy-last-set display double-converts display→kg→display, producing garbled lbs values
- **data-integrity** · `mobile/src/components/StepperLogger.tsx:664` · confidence: high
- lastTopSetDisplay.weight is already in DISPLAY units (set by WorkoutLoggerHost line 977: `const w = unitPref === 'lbs' ? roundToNearestQuarterLb(kgToLbs(ls.weight_kg)) : ls.weight_kg`). ### `workout-volume-wrong` — GET /workouts total_volume_kg computed from SUM(weight_raw) not SUM(weight_raw/8)
- **data-integrity** · `peak-fettle-agents/server/routes/workouts.js:207` · confidence: high
- The query sums weight_raw (kg×8 SMALLINT) values across ALL sets and then divides by 8 once. This produces SUM(kg×8)/8 = SUM(kg), which is the sum of individual weights not total volume. Total volume should be SUM(weight_kg × reps) = SUM((weight_raw/8) × reps). The current formula omits reps entirely, so a user who does 5 sets of 3 reps at 100 kg gets reported as 500 kg volume instead of 1500 kg. This is shown on the workout history list card and likely feeds other analytics.
- **Fix:** Change the aggregate to: `COALESCE(SUM((s.weight_raw / 8.0) * s.reps) FILTER (WHERE s.kind = 'lift' AND s.weight_raw > 0 AND s.reps > 0), 0) AS total_volume_kg`. Also remove the `/ 8.0` scalar divisor at the end.

### `backup-ratelimit-in-process` — Backup rate-limit counter is in-process Map — bypassed by restarts and concurrent instances
- **security** · `peak-fettle-agents/server/routes/backup.js:39` · confidence: high
- The 12-PUT/day rate limit for encrypted backup uploads is tracked in a module-level JavaScript Map. This state is lost on every process restart, and if multiple server instances run (e.g. Railway horizontal scaling), each instance has its own counter. An attacker can bypass the limit by either restarting the process or hitting different instances. Since backups are stored in Supabase storage at `{userId}/backup.json`, unlimited writes could consume significant storage and overwrite a good backup with a corrupt one, causing data loss for the user.
- **Fix:** Move the rate limit counter to Postgres or Redis. A simple approach: INSERT INTO backup_rate_limit (user_id, window_start, write_count) with an ON CONFLICT update, and check/increment atomically. The existing express-rate-limit package used elsewhere in the app can be configured with a Postgres store.

### `direct-add-member-display-name-collision` — POST /groups/:id/members direct-add looks up target user by non-unique display_name
- **correctness** · `peak-fettle-agents/server/routes/groups.js:514` · confidence: high
- display_name is stored in the users table but is not constrained to be unique (no UNIQUE index in any migration file). If two users share a display name (e.g. both set their name to 'Alex'), the query returns whichever row Postgres happens to return first, and the admin adds the wrong person to the group. The victim user is added to a group without consent, and the intended user is not added. There is also a user enumeration surface: an attacker can probe whether a display_name exists on the platform.
- **Fix:** Either (a) enforce a UNIQUE constraint on users.display_name and communicate that names must be unique, or (b) change the add-by-username API to require a user handle or share-link only, removing the display_name lookup path. At minimum, if ambiguity is detected (rows.length > 1), return a 409 conflict rather than silently picking one.

### `math-random-uuid` — genId() uses Math.random — not cryptographically random, UUIDs are predictable
- **security** · `mobile/src/db/localDb.ts:39` · confidence: high
- genId() generates RFC4122 v4 UUIDs using Math.random(), which is not cryptographically secure. In React Native, Math.random() is seeded from a predictable source and its output can be guessed. These IDs are used as primary keys for routines, sets, bodyweight entries, and other personal data. A guessable local ID is not immediately exploitable over the network, but if IDs are ever sent to the server or shared (e.g. group features), an attacker could predict or enumerate them. It also violates the stated intent of 'RFC4122-ish v4 UUID'.
- **Fix:** Use expo-crypto.getRandomValues() or react-native's built-in crypto.getRandomValues(). Example: replace the Math.random call with `const bytes = new Uint8Array(16); crypto.getRandomValues(bytes);` and construct the UUID from those bytes. Alternatively import and call `expo-crypto`'s `randomUUID()` directly.

### `widget-weight-raw-comparison` — widgetBridge PR counting compares raw weight_raw integers, ignoring weight_kg — wrong PRs after v3 migration
- **data-integrity** · `mobile/src/services/widgetBridge.ts:69` · confidence: high
- The SQL query at line 94 only selects `weight_raw` and not `weight_kg`. After the v3 migration, sets written by the v3 path store their precise weight in `weight_kg` and a rounded `weight_raw` (kg×8). For sets where the user entered e.g. 82.3 kg, weight_raw=658 (≈82.25 kg) while weight_kg=82.3. Rows written BEFORE the v3 migration have weight_raw populated but weight_kg=NULL (then backfilled to weight_raw/8.0). The filter `s.weight_raw == null` would skip any set where weight_raw is NULL — but more critically, comparing weight_raw integers directly (rather than COALESCE(weight_kg, weight_raw/8.0)) means the PR count could be wrong if any row has weight_raw=NULL (e.g. a set inserted via a code path that only sets weight_kg). The PR-count displayed on the widget would then be incorrect.
- **Fix:** Change the SQL query to: `SELECT id, exercise_id, reps, COALESCE(weight_kg, CAST(weight_raw AS REAL)/8.0) AS weight_kg_val, logged_at FROM sets WHERE kind='lift' AND logged_at >= ?` and update SetRow + countPRsThisWeek to use weight_kg_val instead of weight_raw. Filter on `weight_kg_val == null` rather than `weight_raw == null`.

### `connector-ignores-weight-kg` — connector.ts ignores sets.weight_kg on upload — Pro users' sets synced with precision loss
- **data-integrity** · `mobile/src/db/connector.ts:129` · confidence: high
- The PowerSync connector's uploadData() reads only weight_raw from the CRUD op data and derives weightKg as weightRaw/8. Since v3 sets store the exact entered kg in weight_kg (e.g. 82.3) and the rounded integer in weight_raw (658 → 82.25 kg), the server receives 82.25 instead of 82.3 kg. This reverses the entire point of the v3 precision fix for Pro users whose sets are synced via PowerSync. The op data from PowerSync includes all columns the schema defines — weight_kg is in the PowerSync schema (schema.ts does NOT include weight_kg, see below), so this is also a cross-file contract break: schema.ts only declares weight_raw in the PowerSync table definition, meaning weight_kg is never synced by PowerSync at all.

Also add weight_kg to the PowerSync schema in schema.ts so PowerSync tracks the exact-precision column: `new Column({ name: 'weight_kg', type: ColumnType.REAL })`.

### `migration-no-transaction` — Migration runner applies statements without a wrapping transaction — partial migration leaves DB in broken state
- **data-integrity** · `mobile/src/db/migrations.ts:151` · confidence: high
- The runner applies each DDL statement individually with no BEGIN/COMMIT wrapper. If the app crashes or is killed mid-migration (e.g. iOS force-quit between the 5th and 6th CREATE TABLE of v2's 22 statements), the user_version is not advanced, so the next launch re-runs the migration from the top. CREATE TABLE IF NOT EXISTS statements are idempotent, but the v3 ALTER TABLE ADD COLUMN is NOT idempotent — running it a second time will fail with 'duplicate column name: weight_kg'. The v3 backfill UPDATE is also not idempotent in the failure case where weight_kg already has some non-NULL values from a partial prior run.
