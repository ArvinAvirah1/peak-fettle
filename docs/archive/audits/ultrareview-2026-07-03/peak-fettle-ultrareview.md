# Peak Fettle — Ultrareview (2026-07-03)

## Executive summary

Peak Fettle's build health measured clean this run: the repo's own parse-sweep passed 339/339 files (mobile/app, mobile/src, peak-fettle-agents/server, and the lifeos companion app), `node --check` passed clean on all 47 server `.js` files, `node mobile/src/db/__tests__/migrations.test.js` passed 13/13, and a `tsc --noEmit` delta came back at 60 errors — below the documented ~85-error baseline and dominated by the same two known categories (expo-router typed-route strings, a chart `Value` type). A dedicated server security pass across all 18 route files found no SQL injection, no IDOR, and correct JWT/OAuth/tier-enforcement handling. At this scale (mobile ~199 TS files / ~67k lines; server ~47 files / ~12.4k lines), that is a genuinely healthy baseline — the codebase is not in a fragile or actively-corrupted state.

That said, the review surfaced a meaningful cluster of real defects concentrated in three areas. First, **safe-area-in-modal is violated in 8 separate screens/components**, including the two highest-traffic surfaces in the app (SetEntryForm — the primary set-logging modal — and the Pro plan-detail sheet), despite this being a named, previously-fixed invariant in project docs; the fix pattern is known and already correctly applied elsewhere in the same files, so this is a consistency failure, not a knowledge gap. Second, a **live server bug breaks constraint deletion for every Pro/synced user** (a UUID sent where the server expects an enum, guaranteed 400). Third, the **warmup-ladder rung filter has an off-by-a-lot magic number bug duplicated identically in both the mobile and server training-engine copies**, silently truncating warm-up sets for lighter/beginner lifters in live plan generation. A secondary, process-level risk cluster involves an env-flag-gated Pro self-upgrade escalation path whose state is leaked pre-auth via `/health`, and two safety-critical modules (the push dispatcher with two prior P0 incidents, and the weight-unit conversion helpers with a documented past incident) that currently have zero automated test coverage.

**Verdict: the codebase is structurally sound and not corrupted, but ships a handful of concrete, fixable, user-facing defects (safe-area, constraint delete, warmup ladder) that should be fixed before the next release, alongside a smaller set of monetization/observability gaps worth closing on a slightly longer horizon.**

## Methodology & scope

Two-stage review: Stage 1 ran six parallel coverage agents across disjoint areas of the codebase (screens/components, data/hooks/db, lib/planGen/services, server security, server cron/trainingEngine/schema, deps/tests/architecture). Stage 2 independently re-verified, from a fresh context, every Critical and High finding plus four flagged Medium candidates, by re-deriving the evidence directly against the repo rather than trusting Stage 1's claims.

**Scope**: `mobile/`, `peak-fettle-agents/`, `db/`. **Out of scope**: `MyApp/`, `marketing-site/`, `mind/`, and other legacy/scaffold folders (`peak-fettle-app/`, `engine-v2-testrun/`, `ollama-agents/`, the nested Qt `Peak Fettle/build/` prototype) — confirmed dead/unimported, noted only as repo-hygiene items below.

**Verification outcome**: all 19 Critical/High items independently verified — 19 confirmed, 0 rejected, 1 severity-reinterpreted (the `.fuse_hidden` artifact: factually confirmed as present, but reclassified from "critical corruption" to a repo-hygiene cleanup item since no code-behavior impact was found). Two "CRITICAL"-labeled test-coverage gaps (push-dispatcher, and by extension the review's framing of untested safety-critical code) are confirmed as real coverage gaps, not live runtime bugs — this nuance is preserved in each finding's write-up below. A small number of raw-file self-retracted false positives (e.g. `scaleDown`'s deep-clone concern, `sequence.ts`'s hard/easy interleave, a `localSchema.ts` "corruption" that was a false alarm) were excluded entirely per the raw files' own re-tracing.

### High

**deleteConstraint sends a UUID where the server expects the constraint_type enum — breaks constraint deletion for every Pro user** — `mobile/src/api/constraints.ts:56-58`, called from `mobile/app/(tabs)/profile.tsx:1051`; server: `peak-fettle-agents/server/routes/constraints.js:80-92` — Problem: the client builds `` `/constraints/${constraint.id}` `` (a UUID), but the server route is `DELETE /:type`, validated against a closed enum (`lower_back, knee, shoulder, ...`). Any UUID fails that check and the server returns 400. Every Pro/synced user's attempt to delete a physical restriction fails; free/local-first users are unaffected (they hit a local SQLite DELETE instead). Fix: change `deleteConstraint` to accept `constraint_type` and update the call site to pass `constraint.constraint_type` instead of `constraint.id`.
```js
export async function deleteConstraint(id: string): Promise<void> {
  await apiClient.delete(`/constraints/${id}`);
}
// server: router.delete('/:type', ...) validates against CONSTRAINT_TYPES enum → 400 invalid_type
```
Provenance: verifier-confirmed.

**SetEntryForm — the primary "log a set" modal — renders its header under the Dynamic Island on notch devices** — `mobile/src/components/SetEntryForm.tsx:14-28` (import), `397-431` (Modal + header) — Problem: `SafeAreaView` is imported from `react-native` core (not `-safe-area-context`), and no `useSafeAreaInsets()`-derived `paddingTop` is applied to the header. This is the highest-traffic modal in the app (opened every time a set is logged), and safe-area does not reliably propagate inside a `<Modal>` per this project's own previously-documented invariant. Fix: import `useSafeAreaInsets` from `react-native-safe-area-context` and apply `paddingTop: Math.max(insets.top, 12)` to the header.
```jsx
import { Modal, ..., SafeAreaView, Alert } from 'react-native';
<SafeAreaView style={[styles.container, ...]}>
  <KeyboardAvoidingView ...><ScrollView ...>
    <View style={styles.header}>
      <Text style={[styles.exerciseName, ...]}>{exercise.name}</Text>
```
Provenance: verifier-confirmed.

**group-streaks.js automatic weekly cron run is not idempotent — a delayed/retried run on a non-Monday silently corrupts the weekly evaluation key** — `peak-fettle-agents/server/cron/group-streaks.js:30-35` (`prevMondayUTC`), `95-108` (`run()`) — Problem: the automatic (no-arg) path uses `prevMondayUTC`, which merely subtracts 7 days from "now" and truncates to midnight — it does not compute the real ISO Monday. This is only correct if invoked exactly on a Monday. A crashed cron retried by an on-call engineer on a different day, or any scheduler misfire, produces a non-Monday `weekStart`, silently creating a misaligned row keyed on `(group_id, week_start)` instead of erroring. The manual CLI path validates Monday-ness and throws; the automatic path has no equivalent guard. Fix: compute the ISO Monday from "now" properly (reuse the file's own `isoWeekStartOf` helper and subtract 7 days), or assert `startedAt` is a Monday and refuse otherwise.
Provenance: verifier-confirmed.

**Weight-unit conversion helpers (units.ts) have zero test coverage** — `mobile/src/constants/units.ts` (`kgToLbs`, `lbsToKg`, `roundToNearestQuarterLb`, `formatWeight`, `displayToKg`, `kgToInputValue`, `parseWeightInput`) — Problem: this is the exact module the project's own documentation flags as its #2 highest-leverage risk (the historical 185lb-stored-as-185kg incident class), and independent re-grep confirmed zero test files reference any of these symbols. **This is a coverage gap, not a demonstrated live bug** — every live call site audited elsewhere in this review correctly used these helpers. **tier-policy.test.js tests only the pure branch function, not real screen/hook call sites** — `mobile/__tests__/tier-policy.test.js:1-14` (whole file) — Problem: the project's documentation calls a screen/hook bypassing `isLocalFirst` and calling raw `api/*` on mount "the #1 recurring bug," yet the only test for this invariant is 5 assertions against the pure `isLocalFirst`/`syncsToServer` functions with literal objects — no integration test exercises `(tabs)/index.tsx`, `profile.tsx`, `rankings.tsx`, etc. to guarantee a free user never fires a personal-data network call on mount. ### Medium

**warmupLadder filters out warm-up rungs below a flat 40kg cutoff instead of the apparently-intended 20kg bar weight — duplicated identically in mobile and server** — `mobile/src/lib/trainingEngine/loading.ts:46-60` and `peak-fettle-agents/server/lib/trainingEngine/loading.js:60-73` (byte-identical logic in both copies) — Problem: the comment states intent as "skip rungs at/below the empty bar (20kg)," but the code filters `weight_kg > barWeight + 20` = `> 40kg`, a flat cutoff independent of the lifter's actual working weight. For a 60-70kg working weight, 2 of the 4 warm-up rungs (40%, 55%) get silently dropped; for a ~50kg working weight, only 1 of 4 survives — exactly the beginner/lighter-lifter population the feature is meant to help. Confirmed live in both the mobile plan-generation path (`loading()` calls it directly) and the server copy it was ported from. Fix: most likely correct fix is `.filter((r) => r.weight_kg > barWeight)` (drop only rungs at/below the empty bar); confirm intended threshold with the founder, then apply identically to both copies, and add a regression test with a sub-100kg working weight.
```js
const barWeight = 20;
const rungs = [0.4, 0.55, 0.7, 0.85];
return rungs.map(...).filter((r) => r.weight_kg > barWeight + 20); // should likely just be `> barWeight`
```
Provenance: verifier-confirmed (one finding covering both mobile and server copies, per de-duplication).

**`/insights/*` routes are authenticated but not paid-gated, letting free users make a personal-data server round trip the local-first architecture says should never happen** — `peak-fettle-agents/server/index.js:169` (mount), `peak-fettle-agents/server/routes/insights.js` (all handlers: `/recovery`, `/readiness`, `/deload`, `/deload/ack`) — Problem: every other Training-Engine-adjacent route (`/plans/generate`, `/plans/:id/regenerate`, `/import/csv`) explicitly re-checks `tier = 'paid'`, and the mobile client ships an on-device fallback (`insightsLocal.ts`) implying these are meant to be Pro-only. As shipped, any authenticated free user can call these endpoints directly (properly scoped by `req.user.id`, so not an IDOR — just a missing tier gate) and get full Training Engine analytics for free, plus incur the exact "free user makes a personal REST call" pattern the project's architecture explicitly forbids. Fix: add `requirePaid` to the `/insights` mount or each handler, unless free access is an intentional product decision (in which case document the exception).
Provenance: verifier-confirmed (one of the four flagged Medium candidates).

**db/schema.sql drops `sets.weight_kg` and never re-adds it — the server's canonical `sets` table is permanently on the lossy `weight_raw`-only representation** — `db/schema.sql:141-168` (initial CREATE TABLE includes `weight_kg NUMERIC(6,2)`), `~1432` (`DROP COLUMN IF EXISTS weight_kg CASCADE`), `4972-4973` (later comment re-affirms the drop) — Problem: running schema.sql top-to-bottom first creates `weight_kg`, then a later folded migration permanently drops it in favor of `weight_raw` (kg×8, lossy 0.125kg resolution); no later section re-adds an exact-kg column. Every downstream percentile/PR function reads `s.weight_raw / 8.0`. This is the server-side mirror image of the mobile app's documented exact-kg invariant — that invariant is explicitly scoped to the on-device schema in project docs, so this is a real, if narrower-than-first-glance, architectural divergence rather than a violation of the letter of the documented rule. **`overallStrengthPercentilePartial`'s subset-DOTS fit has no finite/positive guard on the population fit, unlike every other guard in the same file** — `mobile/src/lib/strengthModelV3.ts:385-401` (subset fit), `231-258` (`fitLognormal`) — Problem: `fitLognormal` computes `sigma = szy / szz` with no guard against a degenerate input; `overallStrengthPercentilePartial` checks the user's own `dots` value for finiteness but never validates that `mu`/`sigma` derived from the population/standards fit are finite before using them. A degenerate fit would silently produce a false "0th percentile" via `clampPct(NaN) → 0` instead of the intended null/no-estimate sentinel — inconsistent with the NaN-sentinel discipline used everywhere else in the file. Fix: add `if (!Number.isFinite(mu) || !Number.isFinite(sigma) || sigma <= 0) return null;` immediately after computing the fit.
Provenance: verifier-confirmed (spot-checked during due diligence, consistent with raw finding).

**Engine v2 `distributeToSessions` per-session set-count rounding can overshoot the computed weekly-volume target by ~20%+, undermining the MEV/MRV safety ceiling** — `mobile/src/lib/trainingEngine/v2/engine.ts:326-354` — Problem: `perSession` is computed via `Math.ceil`, then `nEx`/`setsEach` are each independently rounded with a `Math.max(2, ...)` floor; composing these can make the realized per-session set count exceed the intended `perSession` (concretely: `perSession=5` → `nEx=2`, `setsEach=3` → actual `6`, a 20% overshoot), repeated across every training day for that muscle each week, with no trace/warning outside the unrelated `perSessionCap` branch. This can push a muscle's actual weekly volume past `mrv` even though `allocateVolume` had clamped the intended target into `[mev, mrv]`. Fix: compute `setsEach`/`nEx` so their product never exceeds `perSession` (distribute any remainder deliberately rather than rounding both up), and consider a post-hoc clamp against `mrv` with a trace line if still exceeded.
Provenance: coverage-stage finding, spot-verified during due diligence (confirmed consistent with the described composition of `Math.ceil`/`Math.round`/`Math.max`).

**`parseTargetReps` mis-parses the `carry` pattern's `"30-40m"` (meters) as 30 reps, feeding a bogus rep count into e1RM-based weight math** — `mobile/src/lib/trainingEngine/loading.ts:18-22, 33-37`; confirmed live via `templates.ts` (4 call sites using `reps: '30-40m'`) — Problem: the regex `/(\d+)/` matches the first number group regardless of unit, so a distance-based carry exercise ("30-40 meters") is treated as "30 reps," producing an arbitrary e1RM-derived weight halving that has no relationship to a legitimate carry-loading scheme. Fix: special-case `pattern === 'carry'` (or any non-numeric-rep-range slot) to skip e1RM-based load derivation.
Provenance: coverage-stage finding, spot-verified during due diligence (template literal occurrences confirmed at the cited line numbers).

**Engine v2's `pctForReps(repHi)` silently overrides the hand-tuned `repZones[...][2]` percentage in every GoalSpec, making that field dead data** — `mobile/src/lib/trainingEngine/v2/engine.ts:546-596`, `params.ts` (GoalSpec definitions) — Problem: the load calculation always derives %1RM from the rep range's upper bound via an NSCA lookup table, rather than the manually-curated percentage baked into each zone. **`parseWeightInput` mishandles thousands-separators and repeated commas** — `mobile/src/constants/units.ts:114-120` — Problem: `.replace(',', '.')` has no global flag, so it only replaces the first comma. A comma-decimal-locale input like "1,234" (meant as 1234) silently mis-parses to 1.234 with no validation error. Given the app's weight fields are typically ≤3 digits, real-world blast radius is low, but it is a genuine locale-parsing defect in the exact module flagged as this codebase's #2 highest-risk area. **Local routine/schedule/set tables have no `user_id` scoping in their queries despite the column/index existing** — `mobile/src/data/routines.ts:84-93, 96-110, 142-200`, `mobile/src/data/schedule.ts:137-164`, `mobile/src/data/setEditing.ts:46-86` — Problem: `routines` has a `user_id` column and index, but CRUD queries never filter by it; `schedule` has no `user_id` column at all (single fixed-id row); `sets`/`workouts` CRUD operates by `id` alone.
