### [P1] A2-01 — COSMETIC_TIERS duplicate keys silently override hair-color tiers

- **File:** `mobile/src/components/avatar/peakAvatarOptions.ts:279-343`
- **Problem:** `COSMETIC_TIERS` is a single JS object literal. The keys `teal`, `silver`, `violet`, `rose`, `gold`, and `sky` appear **twice** — first for hair-color unlocks (lines 279–284) and again for accent-theme unlocks (lines 339–343). In a JS object literal, the last definition wins silently: every hair-color tier entry is overridden by its accent-theme counterpart. TSC already flags two of these as `TS1117: An object literal cannot have multiple properties with the same name` (`peakAvatarOptions.ts:340,3` and `343,3` in `00-tsc.txt`). Concretely: `teal` hair color should require `{ streak: 7 }` but its entry is replaced by the accent-theme `teal` entry of `{ streak: 7 }` (same value, so no visible impact here), while `silver` hair color should be `{ streak: 30 }` but accent-theme `silver` is also `{ streak: 30 }` (masked), `violet` hair color should be `'pro'` but accent-theme `violet` is `{ streak: 100 }` — **so `violet` hair color is accessible at 100-day streak instead of requiring Pro**. Similarly, `rose` hair color should be free but the accent-theme entry sets it to `{ streak: 30 }` (but `rose` is not a hair color id, only background/accent, so net impact is only `violet`). The `gold`/`sky` pair have the same value in both sections so no functional regression there, but the structural bug exists. **Actual impact: the `violet` hair color is accessible at 100-day streak for free users, bypassing its `'pro'` gate.**
- **Evidence:**
  ```ts
  // Line 279-284 — Hair colors section:
  teal:             { streak: 7 },
  pink:             { streak: 7 },
  silver:           { streak: 30 },
  platinum:         { streak: 30 },
  violet:           'pro',          // ← overridden below
  skyBlue:          'pro',
  
  // Lines 339-343 — Accent themes section (WINS):
  gold:             { streak: 7 },
  silver:           { streak: 30 },
  rose:             { streak: 30 },
  sky:              { streak: 30 },
  violet:           { streak: 100 }, // ← silently replaces 'pro' above
  ```
- **Invariant/Rubric:** Cosmetics gating — a free user can equip a Pro hair color (violet) at 100-day streak. P1 (wrong behavior visible in normal flows; tier guard bypassed).
- **Suggested direction:** Namespace the accent-theme keys to avoid collision: rename them `accentGold`, `accentSilver`, `accentRose`, `accentSky`, `accentViolet` in both `ACCENT_THEME_IDS` and `COSMETIC_TIERS`, and update `ACCENT_THEME` record keys to match. Keep hair-color keys as-is.
- **Confidence:** HIGH

### [P1] A2-02 — Wristband IDs mismatch COSMETIC_TIERS keys — streak items are free

- **File:** `mobile/src/components/avatar/peakAvatarOptions.ts:245-249` and `317-319`
- **Problem:** `WRISTBANDS_IDS` uses the bare ids `'teal'`, `'gold'`, `'neon'` (lines 247). But `COSMETIC_TIERS` keys these as `'teal_wristband'`, `'gold_wristband'`, `'neon_wristband'` (lines 317-319) to avoid collision with hair colors. At runtime `isUnlocked('teal', ctx)` hits the `teal` key from the hair-color section (or accent section after the duplicate collision in A2-01), and `isUnlocked('neon', ctx)` finds **no entry** — defaulting to `'free'`. Result: the `neon` wristband (intended to require a 30-day streak) is freely available to everyone; `teal` and `gold` wristbands accidentally pick up the streak value from an unrelated category.
- **Evidence:**
  ```ts
  // WRISTBANDS_IDS (line 247):
  'teal', 'gold', 'neon',          // streak unlocks
  
  // COSMETIC_TIERS (lines 317-319):
  teal_wristband:   { streak: 7 },    // keyed with suffix to avoid clash
  gold_wristband:   { streak: 30 },
  neon_wristband:   { streak: 30 },
  // → isUnlocked('neon', ctx) never finds 'neon_wristband'; defaults to 'free'
  ```
- **Invariant/Rubric:** Cosmetics gating bypass. The `neon` wristband is freely equipable by any user regardless of streak.
- **Suggested direction:** Either rename `WRISTBANDS_IDS` entries to use the `_wristband` suffix (`'teal_wristband'`, `'gold_wristband'`, `'neon_wristband'`) to match the tier-map keys, or invert and rename the tier-map keys to match the IDs. Ensure the fix is consistent across `WRISTBANDS_IDS`, `COSMETIC_TIERS`, and any `setEquipped`/`getEquipped` slot keys already persisted in the DB.
- **Confidence:** HIGH

### [P1] A2-03 — AddConstraintModal: SafeAreaView inside Modal does not clear Dynamic Island

- **File:** `mobile/app/(tabs)/profile.tsx:520-622`
- **Problem:** `AddConstraintModal` renders a `<Modal>` and places `<SafeAreaView>` as the immediate child of `<Animated.View>` (line 529). Per Invariant 3 (`AUDITOR_BRIEF.md`), `SafeAreaView` inside a `<Modal>` does not reliably push content below the Dynamic Island on iPhone. The modal header (the "Add Restriction" title row at line 538) has no manual `paddingTop` derived from `useSafeAreaInsets()`, so it can sit under the notch/island on devices with a Dynamic Island. The correct fix is `paddingTop: Math.max(insets.top, 12)` applied directly to the header row.
- **Evidence:**
  ```tsx
  // profile.tsx:528-538
  <Animated.View style={[{ flex: 1 }, sheetAnimStyle]}>
    <SafeAreaView style={[addConstraintStyles.container, ...]}>
      <KeyboardAvoidingView ...>
        <View style={[addConstraintStyles.handle, ...]} />
        {/* Header — no manual insets.top offset */}
        <View style={[addConstraintStyles.header, ...]}>
          <Text ...>Add Restriction</Text>
  ```
- **Invariant/Rubric:** Invariant 3 — `SafeAreaView`/`useSafeAreaInsets()` around or inside an RN `<Modal>` does not push content below the Dynamic Island.
- **Suggested direction:** Remove `<SafeAreaView>`; call `useSafeAreaInsets()` inside `AddConstraintModal`, then apply `paddingTop: Math.max(insets.top, 12)` to the outer `Animated.View` container or directly to `addConstraintStyles.handle`'s top margin.
- **Confidence:** HIGH

### [P1] A2-04 — `useEffect` calls `loadConstraints` before it is defined (stale-closure / hoisting)

- **File:** `mobile/app/(tabs)/profile.tsx:694-697`
- **Problem:** `loadConstraints` is declared with `useCallback` at line 699, but the `useEffect` that calls it is at line 695 — **above** the `useCallback` declaration. In JavaScript, `useCallback` is not hoisted (it's a runtime call, not a function declaration), so on first render the `useEffect` captures the initial value of `loadConstraints`, which is `undefined`. In practice React's render order means `loadConstraints` is defined before the effect fires (effects run after render, by which point the entire function body has executed), so this does not currently crash. However, the `useEffect` has an **empty dependency array `[]`**, which means it never re-runs even when `user` changes (e.g. after login or tier switch). The correct pattern is to add `loadConstraints` to the deps array, which also requires moving its definition above the effect.
- **Evidence:**
  ```ts
  // profile.tsx:694-699
  useEffect(() => {
    loadConstraints();
  }, []);  // ← missing loadConstraints dep; loadConstraints defined below
  
  const loadConstraints = useCallback(async () => {
    ...
    if (isLocalFirst(user)) { ... }
  }, [user]);
  ```
- **Invariant/Rubric:** Missing `useEffect` deps (Rubric P1 — stale deps array); if user logs in or changes tier after mount, constraints are not reloaded.
- **Suggested direction:** Move `loadConstraints` declaration above the `useEffect` and add it to the deps array: `useEffect(() => { loadConstraints(); }, [loadConstraints]);`.
- **Confidence:** HIGH

### [P2] A2-05 — Glossary `scrollToIndex` effect has stale deps (empty array captures initialTerm/filtered at mount)

- **File:** `mobile/app/glossary.tsx:119-125`
- **Problem:** The `useEffect` for deep-link scrolling captures `filtered` and `initialTerm` at mount time (empty dep array, line 125). If `filtered` is empty at the moment the effect runs (e.g. before `GLOSSARY_TERMS` is ready, or a render ordering issue), the early-return fires and the scroll never happens. The comment "run once on mount" is the intent, but `filtered` is derived from `GLOSSARY_TERMS` which is a static import — in practice the list is always populated, so this is a latent risk rather than a current crash. Additionally, `idx > 0` skips the first term (should be `idx >= 0`).
- **Evidence:**
  ```ts
  useEffect(() => {
    if (!initialTerm || filtered.length === 0) return;
    const idx = filtered.findIndex((t) => t.slug === initialTerm);
    if (idx > 0) { // ← should be idx >= 0; skips first term
      setTimeout(() => listRef.current?.scrollToIndex({ index: idx, animated: true }), 300);
    }
  }, []); // run once on mount
  ```
- **Invariant/Rubric:** Stale closure in `useEffect`; off-by-one bug causes deep-link to first glossary term to silently fail.
- **Suggested direction:** Change `idx > 0` to `idx >= 0`; add `[initialTerm, filtered]` to the dep array (or just `[initialTerm]` with `filtered` stabilized by `useMemo`, which it already is).
- **Confidence:** HIGH (off-by-one); MED (stale closure — GLOSSARY_TERMS is static so practically harmless)

### [P2] A2-06 — Profile: `REST_TIMER_PRESETS` is recreated each render (const inside component)

- **File:** `mobile/app/(tabs)/profile.tsx:660`
- **Problem:** `REST_TIMER_PRESETS` is declared with `as const` inside `ProfileScreen()` on line 660. While `as const` is a type-level construct and the value is a tuple literal, it is still evaluated and a new object is allocated on every render of this 1980-line component.   // profile.tsx:637, 660

  const REST_TIMER_PRESETS = [60, 90, 120, 150, 180, 240] as const;

- **Confidence:** HIGH
