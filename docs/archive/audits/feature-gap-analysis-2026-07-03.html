01Where you are ahead — protect these

Before the gaps: five things Peak Fettle ships that the market leaders do not. Every gap-closing decision below should be made in a way that does not erode these.

  • On-device percentile strength rankings, free. strengthModelV3 computes cohort percentiles locally — Hevy, Strong, Fitbod and Boostcamp have nothing comparable at any price. This is your single most marketable stat ("top 8% bench for your weight class") and it currently lives quietly in a tab.
  • Local-first free tier + E2E-encrypted backup with a recovery code. Nobody else does "your training data never touches our servers unless you pay for sync — and even backup is encrypted client-side." In 2026's privacy climate this is a headline, not a footnote.
  • A deterministic, transparent training engine (progressions, deloads, season phases) at a moment when competitors sell black-box "AI trainers." "Every number is explainable" is a real differentiator for serious lifters — and it is also a defensible reason you do not need an LLM to close the adaptive-programming gap (see G15).
  • Streak-credit accountability groups with weekly signals — a social mechanic that creates retention pressure without requiring a content feed, moderation, or public profiles.
  • Machine / pulley mode in the load calculator — most competitors stop at barbell plate math.

03Tier 1 — table-stakes gaps (users switching from Hevy/Strong expect these on day one)

G1 · Exercise demo media + richer detail pages M · 1–2 wks codeL · content

Hevy ✓Fitbod ✓Boostcamp ✓ (form videos)

The single most visible difference when a Hevy user opens Peak Fettle: your ~194-exercise catalog is text-only. Competitors show a looping demo animation, target-muscle diagram, and 2–3 form cues on every exercise. It matters most for beginners (your onboarding funnel) and for trusting engine-suggested swaps (quickSwap hands users exercises they may not know).

Recommended path: skip video entirely. v1 = static muscle diagram (you already ship muscle-map assets for MuscleHeatmap) + 3 written cues per exercise, generated once into the catalog. v2 = licensed animation pack (WebP/Lottie loops, the Alpha Progression approach) for the ~80 highest-frequency movements.

Architecture notes: pure content + UI — no schema migration, no tier branch (catalog is static, not personal data). Free tier is offline, so media must be bundled or lazily cached, watch the app size; a 194 × ~150 KB WebP pack ≈ 30 MB → bundle top-40, cache the rest. No EAS native work.

G2 · Live Activities + Dynamic Island rest timer M · 1–2 wks

Hevy ✓Strong ✓Android: ongoing notification, 2–3 d

Between sets, lifters lock their phone or switch to messages/music. Hevy and Strong keep the rest countdown and next-set target on the lock screen and Dynamic Island; Peak Fettle currently goes silent — the rest timer only lives inside the app. This is the most-felt daily-use gap for anyone coming from those apps, and it makes the app feel more native than almost any other equal-effort feature.

Scope: live rest countdown with +15 s / skip actions, current exercise + next target (from the logger state you already hold), auto-ends with the workout. Android parity via a foreground notification with chronometer.

Architecture notes: iOS ActivityKit requires a native widget-extension target — you already run @bacons/apple-targets in the LifeOS companion, so the pattern (and the plugin-ordering lesson) is in-house. useRestTimer stays the single source of truth; bridge start/update/stop over a small native module. No schema, no server, no tier branch. Needs EAS rebuild → founder push first.

G3 + G4 · Body measurements & progress photos S–M · 3–5 dM · +1 wk photos

Hevy ✓ bothStrong: measurements (paid)Fitbod ✓ both

You track one body metric: weekly bodyweight. The market standard is a measurements module (waist, chest, arms, thighs, body-fat %, custom metrics) with trend charts, plus private progress photos with side-by-side compare. These are top-3 requested features in every tracker's review pile, and for physique-motivated users they are often the retention feature — the training data says "stronger," the tape and photos say "different."

Privacy angle is a gift: "your photos never leave your phone — optionally sealed into your encrypted backup" is a line none of the big competitors can say.

Architecture notes: measurements = one new local table (metric, value, unit, logged_at) → local migration v3→v4 + BACKUP_TABLES registry + additive drift-guarded server table for Pro sync; charts reuse the trends components. Photos = expo-image-picker + app-private storage; keep photos OUT of the default E2E blob (size) behind an explicit "include photos" toggle. Both features tier-branch cleanly (free = local only).

G5 + G6 · Per-set notes & RPE display toggle S · 2–3 d each

Strong ✓ bothHevy ✓ bothPowerlifter must-have

You have workout-level notes (workouts.notes) but nothing per set — where the useful information actually lives ("felt pinchy," "paused reps," "belt on"). Strong's granular set annotations are a top reason serious lifters stay there. Second, you log effort as RIR only; the powerlifting world speaks RPE. Since RPE = 10 − RIR, this is a display-layer toggle, not a data change — keep RIR canonical in the DB and never migrate.

Also cheap in the same pass: set flags (paused / tempo / pin / belt) as a tag byte, searchable later.

Architecture notes: per-set note = one nullable note TEXT column on sets → local migration v4 (additive, safe) + additive server column with the usual 42703 drift guard + backup registry already covers sets. RPE toggle = appSettings flag + logger/engine copy swap. Both fully local-first; zero network.

04Tier 2 — growth & platform gaps (acquisition loops and the iOS ecosystem)

G8 · Shareable workout summary cards S · 3–5 d

Hevy ✓Strong ✓Highest ROI in this report

Every workout that ends in a PR is a free advertisement — if the user can share it. A rendered summary card (volume, PRs, duration, streak, optional percentile flex) exported straight to Instagram stories / iMessage is the cheapest acquisition loop in fitness apps, and you are the only app that can put "Top 8% bench press" on that card — nobody else has the percentile data.

Entirely user-initiated, so it doesn't touch the privacy story: nothing is shared unless the user renders and posts it themselves.

Architecture notes: react-native-view-shot over a styled summary view + expo-sharing share sheet. G7 · HealthKit / Health Connect re-enable M · 1–2 wks + soak

Everyone else ✓Currently stubbed, not absent

services/healthKit.ts has been a stub since 2026-05-28 (iOS 26.5-beta TurboModule crash in react-native-health). The health-metrics screen reports "data unavailable." Every major competitor syncs bodyweight and workouts with Apple Health / Google Fit — and Strong even paywalls it, meaning users literally pay for what you'd ship free.

Recommendation: don't restore react-native-health — swap to the maintained Swift-based @kingstinct/react-native-healthkit (or Expo's healthkit module if it has landed), which is far less likely to throw the NSException that killed boot. Android: Health Connect.

This also un-blocks G13 (watch HR) and auto-bodyweight for the percentile model.

G13 · Apple Watch app XL · 4–8 wks

Hevy ✓Strong ✓Fitbod ✓The biggest single lift in this report

Logging from the wrist (mockup M5 above) — set ticks, rest timer, HR capture — is the #1 App-Store-review differentiator among the top three trackers. It's also the moat feature: hardest to copy, highest perceived seriousness. Your codebase is unusually well prepared: the HealthKit adapter is explicitly documented as the "watch-ready seam," and the LifeOS companion already proved out @bacons/apple-targets native targets + EAS.

Honest sizing: a real watchOS SwiftUI target, WatchConnectivity sync against the local DB, offline set queueing, and battery-sane HR sampling. 4–8 weeks with device testing; do it after G7 (HR comes from HealthKit) and after G2 (Live Activities share the "workout session state" plumbing).

Architecture notes: free-tier watch logging must write to local SQLite via the phone (WatchConnectivity), never REST — the local-first invariant extends to the wrist. Every build iteration = EAS + physical devices; budget calendar time, not just eng time.

G9 + G11 + G12 · Routine share links · competitor importers · group leaderboards SM

Hevy: share links ✓ + Strong-CSV import ✓

Three smaller growth features that compound:

  • Routine share links (M · ~1 wk): "send my routine to a friend" is Hevy's quiet viral loop. Server stores a routine blob behind a short link; deep link imports it. Your import path is already hardened — the DATA-01 allowlist in routineExerciseFields.ts was built exactly for untrusted routine JSON, supersets/dropsets included.
  • Strong/Hevy CSV importers (S · 2–4 d each): you already parse Garmin and Strava CSV. Strong and Hevy exports are simpler (one row per set). An exercise-name fuzzy map to your catalog is the only real work. This is the switcher funnel — Hevy grew partly by importing Strong data.
  • Group leaderboards (M · ~1 wk): weekly volume/streak board inside existing groups only — opt-in, pseudonymous, riding the group weekly-signal path (the one sanctioned free-tier network call). Global public leaderboards: skip; they fight your privacy identity and invite sandbagging.
Architecture notes: share links + leaderboards are server-side additions (drift-tolerant migrations, degrade to empty-200 per house rules). Importers are pure client. None require native builds.

G14 · Social feed (follows, likes, comments) XL · 4–6 wks MVP + ongoing moderation

Hevy's #1 differentiatorStrategic decision, not just engineering

Hevy's feed — post workouts, follow friends, comment — is its moat and the main reason it beats Strong on retention. It is also the feature most at odds with Peak Fettle's identity: a feed requires public profiles, server-stored workout content (for free users too), moderation + reporting (App Store requirement), and a cold-start network. That's not a feature, it's a second product.

Recommendation: don't chase it head-on. Deepen the groups system instead — an opt-in group feed (mockup M8): PRs and completed workouts auto-shared to your 3 groups only, with reactions. 80% of the retention mechanic, 20% of the cost, zero public-content moderation surface, and consistent with "explicit publish = allowed network call" (the group-signal precedent). Ship share cards (G8) + routine links (G9) first; they satisfy most "social" demand without a feed.

Architecture notes (if built): posts/reactions tables + fan-out reads, reporting/blocking UI, push hooks into the existing Expo dispatcher. Free-tier participation = explicit user shares only, never ambient sync — same carve-out class as group weekly signals.

05Tier 3 — emerging & nice-to-have (2026 trends and fast-followers)

G15 · In-session weight suggestions (autoregulation) M · 1–2 wks

"Hevy Trainer auto-adjusts your working weights" is 2026's headline feature — but under the hood it's arithmetic on last-session performance, which your deterministic engine already owns. Extend engine v2 with a next-set/next-session suggestion rule (last e1RM + logged RIR → proposed load, mockup M4's suggestion strip). Per the founder rule, the copy never says "AI" — say "engine rule P2: you hit 80×8 @ RIR 2". Transparency becomes the feature. Deterministic = unit-testable = consistent with the verification gate.

G16 · Fatigue-aware plan adjustment M · 1–2 wks

Fitbod's moat is muscle-freshness programming. You already compute readiness/recovery (ReadinessCard, recovery heatmap) — it just doesn't act. Wire it into plan-adjust: readiness low N days → suggest volume trim or early deload (the engine already models deloads and season phases). All on-device.

G17 · Circuits (3+ exercises) & conditioning timers M · 1–2 wks

superset_group is a string and superset_rounds already goes to 10 — the data model supports circuits today; only the pair-oriented UI (SupersetPairSheet) caps it at 2. Unlock 3+ exercise groups, then add EMOM/AMRAP/interval timer modes for the conditioning crowd. Schema cost: zero.

G18 · Siri / App Intents voice logging M · 1–2 wks

"Hey Siri, log 8 reps at 100 kg." App Intents also unlock interactive widget buttons (start workout / start rest from the lock screen widget you already ship). Native + EAS rebuild. Niche today, but it's where iOS is pushing — and hands-are-chalky is a real gym use case.

G19 · Achievements & badges S–M · ~1 wk

You already have the hard parts: streaks, PR detection, and — uniquely — a cosmetics/avatar system that gives badges something to unlock. A local rule engine over existing tables ("100 workouts → jersey option") closes the gamification gap with Jefit-style apps at near-zero architectural cost.

G20 · Localization L · 2–4 wks, mechanical

Hevy/Strong ship 10+ languages; you ship English. i18n extraction across 35 screens is agent-friendly, mechanical work (Sonnet lane) — but only worth scheduling when you decide to market outside English-speaking countries. Note: units are already locale-aware via constants/units.ts.

G21 · Coach / client platform XL · separate product

Hevy Coach is a second business (coaches program for clients, clients log in the app). Flagging for completeness: it's a revenue expansion, not a feature — do not attempt inside the current app shell.
