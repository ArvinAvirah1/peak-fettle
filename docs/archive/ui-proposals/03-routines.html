1

Working starter-split preview

P0

Fix the data-unwrap bug so tapping a starter split opens a real exercise preview with a "Use this" CTA.

The problem: Tapping any starter split sheet showed "No exercises in this session" — a mismatch between how the template API wraps its exercise array and how the UI reads it (gettemplate-unwrap-mismatch). First-run users see empty screens on every split. The change: Unwrap the correct key on the template response; render exercise rows grouped by day; surface a primary "Use this split" CTA. No exercises are lost — they were present in the response but never read.

Addresses: gettemplate-unwrap-mismatch · Effort: S · Risk: low

2

Splits as cards, not pills

P2

Replace bare text pills with preview cards that surface day count, exercise count, and focus tags.

The problem: Pill labels like "PPL" or "5/3/1" mean nothing to a new user who doesn't already know what they are. There's no way to compare splits without tapping each one. The change: Render each starter split as a small card with the programme name, a goal pill (Strength / Hypertrophy), day count, exercises-per-day, and up to three focus-area tags. Users scan instead of tapping blind.

Addresses: starter-split discoverability · Effort: S · Risk: low

P2

Remove the inline "Create schedule" card; make "+ New" the only primary CTA on the list screen.

The problem: The list screen shows both a "+ New" button in the header and a tappable "Create schedule" card in the list body. 5

Rich routine cards

P2

Upgrade each routine row to a card showing name, scheduled days, exercise count, last-performed date, and a completion progress bar.

The problem: Current rows show only the routine name and a "›" chevron. Users can't see at a glance when they last trained, what days it's scheduled, or how much of the programme they've completed. The change: Cards show day badges (M / T / W…), exercise count, "Last: Xd ago", and a thin bar reflecting weekly consistency (sessions done / sessions scheduled this week). All data already exists in the local DB — this is purely a rendering change.

Addresses: routine-list information density · Effort: S · Risk: low

6

Duplicate = tap + toast + rename

P0

Fix the 400 error when duplicating a starter split; add a "Copied to Yours" toast and an inline rename field so users can personalise immediately.

The problem: Duplicating a starter split produced a 400 Bad Request because the copy was sent with an empty or null UUID (save-as-routine-empty-uuid-400). Users hit an error on the one CTA that should be the fastest path to a working routine. The change: Generate a valid UUID client-side before the API call. On success: animate the new card into the "Yours" section with a toast banner containing a "Rename ›" affordance. The rename field is inline — no navigation required.

Addresses: save-as-routine-empty-uuid-400 · Effort: S · Risk: low

Swipe actions on routine rows

NEW

Add a swipe-left gesture on each routine card that reveals Duplicate, Edit, and Delete actions.

The problem: There is currently no quick action surface on routine rows. To delete or duplicate a routine users must navigate into it and hunt for a menu — or there is no path at all. Editor safe-area and sticky save

P2

Wrap the routine editor in SafeAreaView and pin the Save button to a sticky footer above the home indicator.

The problem: The editor sheet header overlaps the device status bar on notched iPhones and Android fullscreen devices (android-fullscreen-routine-editor). The Save button is also at the bottom of the scroll, invisible until users scroll past all exercises. The change: Add SafeAreaView (or useSafeAreaInsets) around the sheet header; extract Save into a fixed footer bar with the correct bottom inset. The list scrolls freely; Save is always reachable.

9

Drag-to-reorder exercises

NEW

Add a drag handle (⠿) to each exercise row in the editor so users can reorder their programme by feel.

The problem: Exercise order is fixed once added. Changing it requires removing and re-adding exercises manually, which also loses set/rep data. The change: Render a braille-dot handle on the leading edge of each row. Long-press (or press the handle) activates drag mode: the lifted row highlights in teal, others shift to make room. On release the new index is committed to local state; Save persists to DB. Use React Native's DraggableFlatList or the RN Reanimated drag utility already used elsewhere in the app.

Addresses: exercise reorder UX · Effort: M · Risk: low

Inline exercise add

NEW

Open the exercise picker as an inline bottom sheet within the editor rather than navigating away to a separate screen.

The problem: Tapping "Add exercise" currently pushes a new screen, losing the editing context and requiring a back-navigation. On slow connections the push animation can make it feel like the tap didn't register. The change: Present ExercisePicker as a half-height bottom sheet that slides up over the editor. Integrates with the existing sheet system already used for the workout logger.

11

Local-first instant save

P1

Save routines to the local SQLite DB immediately; sync to the server in the background. Free users never see a save failure.

The problem: Routines save via a server REST call (routines-templates-never-local). Free users hit save errors when the server rejects the request or the device is offline. The routine is lost. The change: Write to the local PowerSync DB first (which is instant and always succeeds). Show a "Saved on-device" toast. The sync engine pushes to the server when the user is on a paid plan and has connectivity. This matches the local-first architecture already used for workouts and the decision logged in TICKET-094.

Addresses: routines-templates-never-local · Effort: M · Risk: med (sync conflict logic needed)

13

Per-routine schedule chip

NEW

Show a teal "Mon · Thu" chip on each routine card; tap it to open the schedule editor inline without entering the full routine.

The problem: There's no way to see a routine's scheduled days from the list view. Users must open the routine, scroll to a schedule section (if it exists), and navigate back. Routines with no schedule show nothing — indistinguishable from an unscheduled routine vs. one where scheduling simply hasn't been discovered. The change: Read the routine's scheduled_days field; render a teal pill with abbreviated day names if set, or a dimmed "No schedule" pill if not. Tapping either opens a compact day-picker sheet. The change is read-only for list rendering; the schedule editor already exists.

Addresses: schedule visibility on card · Effort: S · Risk: low

14

Search/filter by goal

NEW

Add a horizontal filter-chip row above the starter splits section so users can narrow by training goal (Strength, Hypertrophy, PPL, Full body).

The problem: As the starter-split library grows, the list becomes unmanageable without any filtering. A new user who knows they want a "full-body programme" has to read every pill/card to find it. The change: Render a scrollable chip row with goal tags pulled from the template metadata (the goal field on each template). "All" is selected by default and shows everything. Selecting a chip filters the list client-side — no network call. If a template has no goal tag it always appears (safe fallback). Works well paired with option 2 (card layout) but is independent of it.

Addresses: starter-split discoverability at scale · Effort: S · Risk: low

Recommended build order

The founder's reported symptoms — "premade routines don't work" and "can't save a routine" — both map to confirmed P0/P1 bugs, not missing features. Fix those first before any visual work.

Option 6 — fix duplicate 400 error (P0)

Option 4 — single "+ New"
