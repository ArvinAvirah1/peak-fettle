# Peak Fettle — Mobile + Server Review (2026-06-12)

**Trigger:** new free account on a real iPhone — nothing works. Home sections spin
forever, "Offline" on 5G, starter routines/save-routine/create-schedule dead,
Rankings 500, profile "Server error" on unit save, survey goal-weight kg-only,
modal headers under the status bar.

**Method:** 7-dimension parallel code review (hooks/local-data, home+workout flow,
routines/schedules, survey/units/prefs, app-wide safe-area sweep, server+migrations,
build-integrity parse sweep) with adversarial per-finding verification. 54 unique
findings; 1 refuted-then-reinstated (see §Reconciliation). Read-only; this doc plus
the two migration drafts are the only files written.

---

## 1. Executive summary — the half-shipped refactor

The 094-A "local-first" refactor is **uncommitted in the working tree**. The
founder's phone runs a build cut from `origin/main` (HEAD = `5ee150a`). HEAD already
contains **Workstream B** — commit `57b19a7` gated PowerSync OFF for free users — but
**not Workstream A**, the hooks that were supposed to read/write on-device SQLite
instead of PowerSync. Net result on the device **today**:

> Free users have **no data source at all**. PowerSync is switched off (hence the
> permanent "Offline" badge on 5G), and the screens still wait on PowerSync watches
> that never emit → Home STREAK / TODAY / RECENT ACTIVITY spin forever. Anything that
> still calls a personal REST endpoint (Rankings, profile save) hits prod — and prod
> is itself drifted, so those 500.

Worse, HEAD commit `5ee150a` **swept 55 staged deletions**: it deleted the 094-B
backup/widget/insights/engine files out from under the app and dropped 6 dependencies
from `package.json`, while `app.json` still references the deleted `@bacons/apple-targets`
plugin. **An EAS build from `origin/main` right now fails at prebuild.** (This is the
exact "commit swept staged deletions" hazard already in project memory.)

And the working-tree fixes are themselves broken: the local DB layer **self-deadlocks
on first use** (a circular `await` in the migration runner), so even after you commit
and rebuild, free users would still hang — just for a different reason.

**So three things are simultaneously true and all must be fixed:**
1. **Device (HEAD) is broken** — half-shipped refactor + drifted prod DB.
2. **Working tree is broken** — localDB deadlock, wrong-path profile PATCH, wrong-table
   health queries, routines never made local, etc.
3. **An EAS build from HEAD won't even compile** — swept deletions + dangling plugin.

The fix is not a patch — it's: repair the working tree, land it as **one atomic commit**,
apply the **ensure-everything** DB migration, rebuild. Details below.

## 3. Reconciliation — the profile "Server error" (why it's both a 404 and a 500)

The two profile findings looked contradictory; they're build-specific:

- **Device today (HEAD `5ee150a`):** `patchProfile` uses the *correct* `/user/profile`.
  The request reaches the server, but the `RETURNING` clause names un-migrated survey
  columns → **500** → client shows "Server error". → fixed by the **ensure-everything migration**.
- **Working tree (next build):** the 094-A edit regressed the path to `/users/profile`
  → **404** → same "Server error" string. → fixed by **reverting the path** + branching
  free users to local `user_profile`.

Both are real. The adversarial verifier that flagged the 500 as "refuted" was reasoning
about the working tree only; on HEAD the 500 is exactly what the founder sees.

## 4. Fix plan — disjoint workstreams (parallel-safe)

Ownership is file-disjoint so these can be implemented in parallel without conflicts.

**A — Local DB core** (`mobile/src/db/localDb.ts`, `migrations.ts`, `__tests__/migrations.test.js`)
- Fix the deadlock: build a raw `handle`-bound shim (`getAll/getFirst/execute` over
  `handle.*Async`) and pass *that* to `runMigrations` + `buildBackupFromDb` — never the
  public singleton. Don't cache a rejected `initPromise` (null it on failure so a retry
  can re-init). Fix the test's hardcoded path to `path.resolve(__dirname, ...)`.
- Verify: `migrations.test.js` 10/10 on Windows; a smoke test that two consecutive
  `localDb.getAll` calls resolve.

**B — Home + workout logging** (`mobile/src/hooks/usePowerSyncLog.ts`, `useWorkout.ts`,
`useStreak.ts`, `mobile/app/(tabs)/index.tsx`, `WorkoutLoggerHost.tsx`)
- Guard `logSet`/`deleteSet` (and `flush`) with `localFirst`; add `localFirst`/`userId`
  to the `useCallback` deps. Make Home's STREAK/TODAY/RECENT read local for free users
  (or render an empty state instead of an infinite spinner). Normalize `exercise_id`
  in `handleSaveAsRoutine` (TICKET-088 pattern). Filter local queries by `user_id`.

**C — Routines / schedules / templates** (`mobile/app/(tabs)/routines.tsx`,
`mobile/src/api/templates.ts`, new `mobile/src/data/routines.ts`, `ScheduleEditorSheet.tsx`,
`RoutineEditorSheet.tsx`, `templates.tsx`)
- Fix `getTemplate` unwrap (P0). Add a tier-branched routines data module: free →
  localDb `routines` CRUD; Pro → existing REST. Starter-split duplicate for free = local
  INSERT. Seed/read local template tables or keep template *browsing* as global REST with
  a real empty/offline state.

**D — Survey / units / prefs** (`mobile/app/onboarding.tsx`, `training-survey.tsx`,
`mobile/app/(tabs)/profile.tsx`, `mobile/src/api/user.ts`, new `mobile/src/data/profile.ts`)
- Revert the PATCH path to `/user/profile`. Add a tier-branched profile writer (free →
  local `user_profile` upsert + `updateUser()`; Pro → `patchProfile`). Add a kg/lb toggle
  to goal-weight (store canonical kg, accept/display lbs) in both onboarding and survey,
  honoring `unit_pref`. Surface save failures (no silent revert / fake success).

**E — Safe-area sweep** (mechanical, many files — see P2/P3 lists)
- Establish one helper (the `ScreenLayout` `SafeAreaView edges={['top','bottom']}` pattern)
  and apply to every full-screen Modal / pageSheet header + every bottom-button footer.
  Use `react-native-safe-area-context` (works on iOS *and* Android), not RN's `SafeAreaView`.
  Set Android `StatusBar` translucent handling once, globally.

**F — Server + migrations** (`peak-fettle-agents/server/routes/percentile.js`,
`user.js`, the two new migration files)
- ✅ **Written:** `migrations/20260612_ensure_schema_complete.sql` (idempotent;
  unbricks the Rankings 500 + profile 500 by re-asserting every route-referenced
  column/table). ✅ **Written:** `migrations/20260612_purge_free_user_training_data.sql`
  (founder-gated; preview counts → transaction → guardrail against wiping everyone if
  `tier` is unpopulated).
- Code: make GET `/percentile` and PATCH `/user/profile` resilient — build `RETURNING`
  from updated columns; return empty rankings rather than 500 when optional tables are bare.

**G — Build hygiene & cleanup**
- Restore the swept 094-B files + deps as ONE atomic commit (verify `app.json`'s
  `@bacons/apple-targets` resolves post-commit). Run the `peak-fettle-verify` parse sweep.
  Delete repo-root junk (`.rmtest`, `CMakeLists.txt`, `qml/`, `resources/`, `src/`,
  `MyApp/`, `peak-fettle-app/`) — confirm none is referenced before deleting.

## 5. Deployment order for the founder

1. **Back up prod** (`pg_dump`), verify restorable.
2. **Apply `20260612_ensure_schema_complete.sql`** to Railway prod. This alone should kill
   the Rankings 500 and the profile "Server error" **on the current build** (no app change
   needed) — quickest win.
3. **Fix the working tree** (workstreams A–E, G) and land as **one atomic commit**; run the
   parse sweep; inspect the staged-`D` column before committing (per project memory).
4. **Push to origin/main** (Railway auto-deploys the server; EAS reads the remote).
5. **Cut a new EAS build** — only now do the local-first reads/writes reach the device.
6. **Verify on device:** new free account → Home loads, log a set, save a routine, create a
   schedule, change units to lb, survey goal-weight in lb — all offline.
7. **Only after on-device verification:** run `20260612_purge_free_user_training_data.sql`
   (preview counts first) and, separately, the founder-gated `20260612_drop_percentile_rankings.sql`.

**Cannot be fixed without a rebuild:** every WT/US/GAP item. The migration (step 2) is the
only thing that helps the *current* build.

## 6. Refuted / open questions

- **Refuted as a *working-tree* claim, reinstated as a *prod* one:** the profile 500 (see §3).
- **Open:** exact prod schema state is unknown — the ensure-everything migration is written to
  be safe either way (idempotent). Recommend running it regardless.
- **Open:** whether to seed local `workout_templates` (offline starter splits) or keep template
  browsing as global REST. Recommendation: keep browsing REST (templates are non-personal),
  but give it a real empty/offline state.
- **Open:** `weight_raw` vs `weight_kg` in prod `sets` — the percentile route assumes the
  integer-encoded `weight_raw`; the ensure migration adds the column if missing, but if prod
  stores data in `weight_kg` only, the Epley estimate will read NULL until a backfill. Low
  urgency (free users compute on-device; Pro estimates degrade gracefully to confirmed-1RM).
